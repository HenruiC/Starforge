# 《天赋猎人》统一战斗角色框架设计文档

版本：V1.0
目标：将所有战斗单位（玩家、敌人、Boss、队友）收敛到统一基类，实现属性/技能/Buff 系统复用，并建立可配置的怪物 AI 框架

---

## 一、系统概述

### 1.1 核心概念

当前 `player.gd` 和 `enemy.gd` 是两套完全独立的 `CharacterBody2D` 子类，导致：
- 每个新角色/敌人类型都要从头写
- 技能系统只能给玩家用
- 属性散落在各自 `@export` 中，无法统一被 Buff 修改
- 敌人 AI 硬编码在 `_physics_process`，没有行为模式概念

**统一战框架的核心思想**：所有"能打架的东西"都是 `CombatUnit`。玩家只是 `CombatUnit + PlayerController`，敌人只是 `CombatUnit + AIController`。

```
                 CharacterBody2D
                       |
                   CombatUnit
                  /    |    \
           Player   Enemy   Boss  (都是 CombatUnit 的不同配置)
           
           控制器决定"谁来操控":
           PlayerController  →  WASD + 鼠标
           AIController      →  行为树/状态机
```

### 1.2 设计目标

| 维度 | 现状 | 目标 |
|------|------|------|
| 角色定义 | Player.gd / Enemy.gd 各自独立 | 统一 CombatUnit + 数据配置 |
| 属性系统 | @export 散落 + 直接读写 | StatsComponent，支持 Buff 叠加 |
| 技能系统 | SkillManager 绑定 Player | SkillComponent，任何 CombatUnit 可挂载 |
| AI 行为 | direction_to 直线追踪 | 行为模式枚举 + 可配置参数 + NavigationAgent2D |
| 新增角色 | 复制粘贴改参数 | 创建 .tres 配置文件 |
| PK 模式 | 不支持 | 两个 CombatUnit 互相释放技能对打 |

### 1.3 在整体游戏中的定位

```
┌──────────────────────────────────────────────────────┐
│                     GameManager                       │
│  (波次管理/敌人生成/UI控制 — 不再直接读写角色属性)      │
└──────────┬──────────────────────────┬────────────────┘
           │                          │
    ┌──────▼──────┐           ┌──────▼──────┐
    │  CombatUnit  │           │  CombatUnit  │
    │  (Player)    │           │  (Enemy)     │
    │              │           │              │
    │ PlayerCtrl   │           │ AICtrl       │
    │ StatsComp    │           │ StatsComp    │
    │ SkillComp    │           │ SkillComp    │
    │ BuffComp     │           │ BuffComp     │
    └──────────────┘           └──────────────┘
           │                          │
           └──────────┬───────────────┘
                      │
              ┌───────▼───────┐
              │   EventBus    │
              │  (通用事件)    │
              └───────────────┘
```

## 二、核心架构

### 2.1 CombatUnit（统一角色基类）

`CombatUnit` 继承 `CharacterBody2D`，是所有战斗角色的基类。它**不包含任何 AI 或输入逻辑**，只包含通用战斗能力。

```
CombatUnit (extends CharacterBody2D)
├── [核心属性]
│   ├── unit_id: String               # 唯一标识
│   ├── team: Team                    # 阵营枚举 (FRIENDLY, ENEMY, NEUTRAL)
│   └── is_dead: bool                 # 死亡标记
│
├── [组件引用]
│   ├── stats: StatsComponent         # 属性系统
│   ├── skills: SkillComponent        # 技能系统 (可选)
│   ├── buffs: BuffComponent          # Buff/Debuff (可选)
│   └── controller: Node              # 控制器 (PlayerController 或 AIController)
│
├── [视觉节点引用]
│   ├── sprite: ColorRect / Sprite2D
│   ├── hit_flash: ColorRect
│   ├── hp_bar: ProgressBar
│   └── attack_area: Area2D
│
├── [通用战斗方法]
│   ├── take_damage(amount, source)   # 受击流程
│   ├── heal(amount)                  # 治疗流程
│   ├── knockback(force)             # 击退
│   ├── _die()                        # 死亡流程 (虚方法)
│   └── get_stat(stat_name) -> float  # 获取最终属性值
│
├── [生命周期信号]
│   ├── damage_taken(amount, current_hp)
│   ├── healed(amount, current_hp)
│   ├── died()
│   └── stat_changed(stat_name, new_value)
```

### 2.2 控制器模式

**不在 CombatUnit 里写输入或 AI 逻辑**。控制权交给独立 Controller：

```gdscript
# PlayerController — 玩家控制
class_name PlayerController
extends Node

var unit: CombatUnit

func _physics_process(delta):
    # 读取 Input，设置 unit.velocity
    var dir := Input.get_vector(...)
    unit.velocity = dir * unit.get_stat("move_speed")
    unit.move_and_slide()
    # 更新瞄准方向
    unit.aim_direction = (get_global_mouse_position() - unit.global_position).normalized()

# AIController — AI 控制
class_name AIController
extends Node

@export var config: AIBehaviorConfig  # .tres 配置

var unit: CombatUnit
var current_state: AIState
var target: CombatUnit

func _physics_process(delta):
    current_state.tick(delta)  # 状态机驱动
    # 状态机内部设置 unit.velocity、触发技能等
```

**为什么用组合而非继承**：同一个 CombatUnit 可以切换控制器（比如眩晕时切到"不可控"状态，队友 AI 接管时从 PlayerController 切到 AIController）。

### 2.3 阵营系统

```gdscript
enum Team {
    FRIENDLY,   # 玩家 + 队友
    ENEMY,      # 怪物 + Boss
    NEUTRAL,    # 可破坏物、中立生物
}
```

`CombatUnit` 的 `team` 属性决定：
- 技能目标筛选（`attack_area` 里过滤 `team != my_team`）
- 碰撞伤害判定
- 友军技能效果（治疗/增益不奶敌人）

## 三、子系统详细设计

### 3.1 StatsComponent（属性系统）

#### 3.1.1 设计目标

核心诉求：**属性值可以被 Buff 临时修改，且修改后自动通知 UI 更新**。

#### 3.1.2 数据结构

```gdscript
class_name StatsComponent
extends Node

## 基础属性（从 .tres 加载，运行时不变）
var _base: StatsResource

## 当前 HP（独立管理，因为变化最频繁）
var current_hp: float:
    set(v):
        var old := _hp
        _hp = clampf(v, 0.0, get_stat("max_health"))
        if _hp != old:
            stat_changed.emit("current_hp", _hp)

## 活跃的属性修改器列表
var _modifiers: Array[StatModifier] = []

signal stat_changed(stat_name: String, new_value: float)
```

#### 3.1.3 StatModifier（修改器）

```gdscript
class_name StatModifier
extends RefCounted

enum ModType { FLAT, PERCENT_ADD, PERCENT_MUL }
enum StackRule { STACK, REFRESH, INDEPENDENT }

var stat_name: String          # 修改哪个属性
var mod_type: ModType          # 修改方式
var value: float               # 修改值
var duration: float            # 持续时间 (0 = 永久)
var remaining: float           # 剩余时间
var source_id: String          # 来源 (buff_id), 用于去重/叠层
var stack_rule: StackRule      # 叠层规则
```

#### 3.1.4 属性计算公式

```
final_stat = (base + sum(FLAT)) * (1.0 + sum(PERCENT_ADD)) * product(PERCENT_MUL)
```

示例：玩家基础攻击力 15，身上有一个 "+10 攻击力" Buff 和一个 "攻击力+20%" Buff：
```
final_attack = (15 + 10) * (1.0 + 0.20) = 30.0
```

#### 3.1.5 标准属性列表

| 属性名 | 类型 | 说明 |
|--------|------|------|
| `max_health` | float | 最大生命值 |
| `move_speed` | float | 移动速度 |
| `attack_power` | float | 攻击力（技能伤害基准） |
| `defense` | float | 防御力（减少受到的伤害） |
| `attack_range` | float | 攻击范围半径 |
| `attack_speed` | float | 攻击速度倍率（乘在 CD 上） |
| `crit_chance` | float | 暴击率 (0.0~1.0) |
| `crit_damage` | float | 暴击伤害倍率（默认 1.5） |
| `projectile_speed` | float | 投射物速度倍率 |
| `knockback_resist` | float | 击退抗性 (0.0~1.0) |

#### 3.1.6 StatsResource（.tres 配置）

```gdscript
class_name StatsResource
extends Resource

@export var max_health: float = 100.0
@export var move_speed: float = 300.0
@export var attack_power: float = 15.0
@export var defense: float = 2.0
@export var attack_range: float = 120.0
@export var attack_speed: float = 1.0
@export var crit_chance: float = 0.05
@export var crit_damage: float = 1.5
@export var projectile_speed: float = 1.0
@export var knockback_resist: float = 0.0
```

### 3.2 SkillComponent（技能系统）

#### 3.2.1 改造目标

当前 `SkillManager` 和所有 `SkillBase` 子类硬编码了 `player` 变量。需要改为通用 `owner: CombatUnit`。

#### 3.2.2 SkillBase 改造

```gdscript
class_name SkillBase
extends Node

## 改造前：var player: CharacterBody2D = null
## 改造后：
var owner_unit: CombatUnit = null       # 技能拥有者
var effect_parent: Node2D = null        # 特效挂载点
var attack_area: Area2D = null          # 目标检测区域

func setup(unit: CombatUnit, ep: Node2D) -> void:
    owner_unit = unit
    effect_parent = ep

func can_execute() -> bool:
    return owner_unit != null and not owner_unit.is_dead

func execute() -> void:
    pass  # 子类实现

## 子类中获取目标的方式改为通过 owner_unit 的 team 过滤
func _get_valid_targets() -> Array[Node2D]:
    var targets: Array[Node2D] = []
    if attack_area == null or owner_unit == null:
        return targets
    for body in attack_area.get_overlapping_bodies():
        if body is CombatUnit and body.team != owner_unit.team and not body.is_dead:
            targets.append(body)
    return targets
```

#### 3.2.3 SkillComponent（替代 SkillManager）

```gdscript
class_name SkillComponent
extends Node

var owner_unit: CombatUnit
var skills: Array[SkillBase] = []

func init(unit: CombatUnit, effect_parent: Node2D, area: Area2D, skill_ids: Array) -> void:
    owner_unit = unit
    for skill_id in skill_ids:
        var skill := SkillFactory.create(skill_id)  # 工厂方法
        if skill:
            skill.setup(unit, effect_parent)
            skill.attack_area = area
            skills.append(skill)
            add_child(skill)

func process_all(delta: float) -> void:
    for s in skills:
        s._process(delta)
        s.try_execute()

## 外部触发（AI 控制器用）
func try_execute_skill(index: int) -> bool:
    if index < 0 or index >= skills.size():
        return false
    return skills[index].try_execute()
```

#### 3.2.4 目标筛选的通用化

当前 `attack_area` 用 `is_in_group("enemy")` 筛选目标。改为用 `team` 属性：

```gdscript
## 旧：
if body.is_in_group("enemy") and not body.is_dead:
    ...

## 新：
if body is CombatUnit and body.team != owner_unit.team and not body.is_dead:
    ...
```

这样 PK 模式下两个 CombatUnit（team 不同）自然就能互相伤害。

### 3.3 BuffComponent（Buff/Debuff 系统）

#### 3.3.1 核心概念

Buff 是一个**有时限的属性修改器容器**。一个 Buff 可以同时修改多个属性。

```gdscript
class_name Buff
extends RefCounted

var buff_id: String
var buff_name: String
var duration: float              # 总持续时间（0 = 无限）
var remaining: float             # 剩余时间
var modifiers: Array[StatModifier]  # 该 Buff 包含的属性修改
var tick_interval: float         # tick 间隔（如 DOT 每 0.5s 触发）
var tick_damage: float           # DOT 伤害
var source_unit: CombatUnit      # 施加者
var icon: String                 # UI 图标
var is_debuff: bool              # 是否为负面效果
```

#### 3.3.2 BuffComponent API

```gdscript
class_name BuffComponent
extends Node

var owner_unit: CombatUnit
var _active_buffs: Array[Buff] = []

func apply(buff: Buff) -> void:
    # 1. 检查叠层规则
    # 2. 应用 modifiers 到 owner_unit.stats
    # 3. 加入 _active_buffs

func remove(buff_id: String) -> void:
    # 1. 移除 modifiers
    # 2. 从 _active_buffs 移除

func tick(delta: float) -> void:
    # 1. 所有 buff 的 remaining -= delta
    # 2. 过期 buff 自动 remove
    # 3. 触发 tick_interval 伤害/治疗

func has_buff(buff_id: String) -> bool
func get_buff_count(buff_id: String) -> int
```

#### 3.3.3 与 StatsComponent 的耦合

BuffComponent 通过 StatsComponent 的 `add_modifier` / `remove_modifier` 接口修改属性。两者解耦：

```
BuffComponent.apply() → StatsComponent.add_modifier(StatModifier)
BuffComponent.remove() → StatsComponent.remove_modifier(StatModifier)
StatsComponent 不关心 modifier 来自哪里
```

### 3.4 AI 控制器框架

#### 3.4.1 行为模式枚举

```gdscript
enum AIBehaviorMode {
    IDLE,       # 原地站立，不做任何事
    PATROL,     # 沿路点巡逻
    GUARD,      # 守卫某点，敌人进入范围后追击，脱战后返回
    CHASE,      # 主动追击目标
    FLEE,       # 远离目标（低血量/恐惧）
    BERSERK,    # 狂暴模式（无视一切，疯狂攻击）
}
```

#### 3.4.2 AI 状态机

```

         ┌─────────→│  PATROL  │←─────────┐
         │          └────┬─────┘          │
         │               │ 发现目标        │
         │          ┌────▼─────┐          │
         │          │  CHASE   │          │
         │          └──┬───┬───┘          │
         │   进入射程  │   │  HP 过低       │
         │    ┌───────▼┐  └──────┐        │
         │    │ ATTACK │         │        │
         │    └───┬────┘    ┌────▼───┐    │
         │   目标死亡│       │  FLEE  │    │
         │    ┌──────┘      └───┬────┘    │
         │    │             脱战 │         │
         │    └────────┐    ┌───┘         │
         │             ▼    ▼             │

         └──────────│  RETURN  │──────────┘
        (回到巡逻/守卫)└──────────┘
```

#### 3.4.3 技能使用条件

AI 使用技能不再是"冷却好了就用"，而是可配置的条件触发：

```gdscript
enum SkillCondition {
    NEVER,                  # 从不使用
    ALWAYS,                 # 冷却好了就用
    TARGET_IN_RANGE,        # 目标在指定距离内
    SELF_HP_BELOW,          # 自身 HP 低于百分比
    TARGET_HP_BELOW,        # 目标 HP 低于百分比
    ALLY_COUNT_BELOW,       # 附近友军数量低于阈值
    TARGET_COUNT_AROUND,    # 周围目标数量达到阈值
    PERIODIC,               # 每 N 秒使用一次
    ON_SPAWN,               # 生成时使用
    ON_DEATH,               # 死亡时使用（亡语技能）
}
```

#### 3.4.4 AIBehaviorConfig（.tres 配置资源）

```gdscript
class_name AIBehaviorConfig
extends Resource

@export var behavior_mode: AIBehaviorMode = AIBehaviorMode.CHASE
@export var detection_range: float = 500.0          # 发现范围
@export var attack_range_preferred: float = 120.0   # 期望攻击距离
@export var flee_hp_threshold: float = 0.25         # 逃跑血量阈值
@export var chase_speed_mult: float = 1.0           # 追击速度倍率
@export var return_range: float = 700.0             # 脱战返回距离（GUARD 模式）
@export var patrol_points: Array[Vector2] = []      # 巡逻路点
@export var guard_position: Vector2 = Vector2.ZERO  # 守卫点
@export var guard_radius: float = 300.0             # 守卫范围
@export var skill_triggers: Array[SkillTrigger] = [] # 技能触发配置
```

#### 3.4.5 SkillTrigger 子资源

```gdscript
class_name SkillTrigger
extends Resource

@export var skill_slot: int = 0            # 使用技能槽位 0/1/2
@export var condition: SkillCondition      # 触发条件
@export var condition_value: float = 0.0   # 条件参数（距离/HP%/时间间隔等）
@export var cooldown_override: float = 0.0 # 覆盖技能 CD（0 = 使用技能自身 CD）
```

#### 3.4.6 寻路集成

`AIController` 使用 `NavigationAgent2D` 替代 `direction_to`：

```gdscript
func _setup_navigation(unit: CombatUnit) -> void:
    var nav_agent := NavigationAgent2D.new()
    nav_agent.path_desired_distance = 8.0
    nav_agent.target_desired_distance = attack_range_preferred * 0.8
    unit.add_child(nav_agent)
    _nav_agent = nav_agent

func _chase_behavior(delta: float) -> void:
    if target == null: return
    
    _nav_agent.target_position = target.global_position
    
    if _nav_agent.is_navigation_finished():
        return  # 已到达
    
    var next_point := _nav_agent.get_next_path_position()
    var dir := unit.global_position.direction_to(next_point)
    unit.velocity = dir * unit.get_stat("move_speed") * chase_speed_mult
    unit.move_and_slide()
```

> **注**：NavigationAgent2D 需要场景中存在 NavigationRegion2D 且已烘焙。当前项目已有 `NavManager` 进行了基础烘焙，可直接利用。

## 五、迁移方案（分 4 阶段）

### 5.1 迁移总则

- **每个阶段独立可测**，不出现"代码改了一半跑不起来"的情况
- **新增代码优先**，不动现有逻辑
- **旧接口保留**，新接口并行运行，确认无问题后删除旧接口
- **场景文件不动**，只在脚本层面改造

### 5.2 阶段一：基础设施搭建（0 风险，纯新增）

**目标**：创建所有新类，不与现有代码发生任何关系。

新建文件清单：
```
scripts/combat/
├── combat_unit.gd           # CombatUnit 基类
├── stats_component.gd       # StatsComponent
├── stats_resource.gd        # StatsResource (Resource)
├── stat_modifier.gd         # StatModifier (RefCounted)
├── buff.gd                  # Buff (RefCounted)
├── buff_component.gd        # BuffComponent
```

### 5.3 阶段二：SkillBase 改造（低风险，接口变更）

**目标**：将 `SkillBase` 的 `player` 改为 `owner_unit`，所有技能子类同步改造。

改造清单：

1. **SkillBase.gd**：
   - `var player: CharacterBody2D` → `var owner_unit: CombatUnit`
   - `func setup(p, ep)` → `func setup(unit: CombatUnit, ep)`
   - 保留 `var player` 作为兼容属性（getter 代理到 `owner_unit`），运行时检查类型
   
2. **所有 SkillBase 子类**（7 个技能文件）：
   - 所有 `player.xxx` → `owner_unit.xxx`
   - `is_in_group("enemy")` → `body.team != owner_unit.team`
   - `player.global_position` → `owner_unit.global_position`

3. **SkillManager.gd**：
   - 保留现有接口，内部改为调用 `SkillComponent`
   - `player` 引用改为 `owner_unit`

改造后的兼容层：
```gdscript
## SkillBase.gd 兼容属性（阶段三完成后删除）
var player: CharacterBody2D:
    get: return owner_unit as CharacterBody2D

```

### 5.4 阶段三：Player 迁移到 CombatUnit（中风险，核心变更）

**目标**：`Player` 改为继承 `CombatUnit`，将通用逻辑上提到基类。

具体步骤：

1. `player.gd`：`extends CharacterBody2D` → `extends CombatUnit`
2. 将 Player 中的通用方法移到 CombatUnit：
   - `take_damage()` → CombatUnit（保留 hit_flash + 死亡流程）
   - `_die()` → CombatUnit（改为虚方法，Player 可覆盖）
   - `knockback()` → CombatUnit
   - `_update_hp()` → CombatUnit
   - 属性相关 → 委托给 `stats: StatsComponent`
3. `player.tscn` 场景不变，脚本引用自动更新
4. `GameManager` 中直接访问 `player.xxx` 的地方改为通过 `player.get_stat("xxx")` 或 `player.stats.xxx`

### 5.5 阶段四：Enemy 迁移 + AI 接入（核心变更）

**目标**：`Enemy` 改为继承 `CombatUnit`，挂载 `AIController`。

具体步骤：

1. `enemy.gd`：`extends CharacterBody2D` → `extends CombatUnit`
2. Enemy 现有属性（`max_health`、`move_speed` 等）→ 移到 `StatsResource` 配置
3. Enemy 现有行为（`_melee_behavior`、`_ranged_behavior`）→ 移到 `AIController`

5. `is_elite`、`is_boss`、`is_ranged` → 改为加载不同的 `AIBehaviorConfig` 和 `StatsResource`
6. `GameManager._spawn_enemy()` → 改为从 `.tres` 配置实例化

## 七、边界情况与 FAQ

### 7.1 边界情况

| 场景 | 处理方式 |
|------|----------|
| 技能释放瞬间施法者死亡 | SkillBase.execute() 开头检查 `owner_unit.is_dead`，已死亡则 return |
| Buff 在死亡后残留 | BuffComponent 在 `_die()` 时 `clear_all()` |
| AI 目标死亡但攻击动画未结束 | ATTACK 状态退出时检查 `target.is_dead`，切换到 IDLE/PATROL |
| 同一 Buff 叠加 | 由 `StackRule` 配置决定：REFRESH（刷新时间）或 STACK（叠层）或 INDEPENDENT（独立存在） |
| NavigationAgent2D 未烘焙时 | AIController 回退到 `direction_to` 直线移动（降级策略） |
| 玩家和敌人的技能效果相同 | SkillBase 的 VFX 逻辑复用，不区分 owner 类型 |
| 两个 CombatUnit team 相同 | 技能/攻击自动跳过（`team != owner_unit.team` 过滤） |

### 7.2 FAQ

**Q: 为什么不把 CombatUnit 做成纯组件（ECS）？**

A: Godot 的节点树天然是组合式的。把 Stats、Skills、Buffs 做成子 Node 已经是组件模式。过度拆分（比如血量做成单独 Node）会增加帧间查询开销，且与 Godot 的场景工作流背道而驰。当前粒度：CombatUnit 带 3~4 个组件 Node，已经足够解耦。

**Q: PlayerController 和 AIController 能否共存？**

A: 可以。比如"魅惑"效果：给敌人临时挂一个 PlayerController（或受限的 AI），原来的 AIController 暂时禁用。Controller 只是一个设置 velocity 和调用技能的节点，随时可以替换。

**Q: 现有的 7 个技能文件需要全部重写吗？**

A: 不需要重写逻辑。只需全局替换 `player` → `owner_unit`，以及 `is_in_group("enemy")` → `team` 检查。VFX 逻辑、伤害计算全都不变。

**Q: 迁移过程中能保证游戏一直可运行吗？**

A: 阶段一完全不动现有代码。阶段二中 SkillBase 保留兼容属性。阶段三 Player 改继承，实际行为不变。阶段四 Enemy 改继承，实测验证。每个阶段都是独立可测的 checkpoint。

**Q: 这个框架能支撑 PK 模式吗？**

A: 能。PK 就是两个 CombatUnit（team 不同）放在同一场景，各自挂不同 Controller。技能目标过滤基于 team，自动互相伤害。唯一需要新增的是 PK 专用的胜负判定和 UI。
