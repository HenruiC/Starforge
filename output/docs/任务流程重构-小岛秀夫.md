## 序幕：现在的体验，像一个电话打错了时间

想象一个场景：

你在教室走廊里清完了第十五只变异体，穿过连廊向北，体育馆的轮廓在暗角中浮现。你握着键盘的手指微紧——前面就是 Boss 了。然后屏幕冻结。一个对话框弹出来——

**"30秒...我活下来了。这些怪物从哪来的？"**

那是你在五分钟前杀完第一波怪物时应该出现的台词。它迟到了整整两个 Stage。像是一封寄错了时间的信。

这不是代码有 bug。这是**信号的弧线走岔了**。

`_trigger_stage_dialogue()` 在 `game_manager.gd:878`——它用 `match stage` 硬编码了对话内容。但这个函数被调用的时候，`stage` 参数的来源已经是正确的。问题不在参数——在于**说这句话的时机**。当 Stage 2 完成、Stage 3 激活的那一瞬间，旧对话和新 UI 在同一个帧里挤了出来。哪个先到？不是看谁先发出信号——是看谁先抢到 SceneTree 的帧。

这篇文章要做的事：把错位的对话放回对的时刻，把模糊的任务流程写成清晰的章节标题，把 Boss 的终结从一个信号变成一个仪式。

## 第一章：对话触发 —— 诊断与修复

### 1.1 症结所在

整个对话系统当前有三层剥离：

| 层 | 位置 | 当前行为 | 问题 |
|-----|------|---------|------|
| **数据层** | `mission_manager.gd:235-238` | Stage 2 的 `dialogue_messages` 已正确定义 | Stage 1 和 Stage 3 **没有**定义 `dialogue_messages`，数据不完整 |
| **触发层** | `game_manager.gd:878-898` | `_trigger_stage_dialogue(stage)` 用 `match stage` 硬编码消息 | 对话内容和 Stage 数据定义**重复**，且不同步 |
| **时序层** | `game_manager.gd:844-856` | `_on_stage_cleared` 调用 `_trigger_stage_dialogue` 但不 `await` | 异步对话未完成就执行了 `_refresh_prompt_ui_from_stage()`，导致新 Stage UI 先行显示 |

**时序层的竞态条件（Race Condition）具体流程：**

```
1. MissionTriggerManager._check_stage_completion()
   → stage.status = COMPLETED
   → stage_completed.emit(stage.stage_id)      ← 信号发出
   → _try_advance_to_next()
   → 如果 player 已在 zone_gym_entrance → _activate_stage(2)  ← Stage 3 立即激活
   
2. GameManager._on_stage_cleared(cleared_stage_id)  ← 处理 Stage 2 完成信号
   → CombatFeedback.screen_shake(8.0)           ← 屏幕震动，立即执行
   → _trigger_stage_dialogue(2)                 ← 启动异步对话（游戏暂停）
   → _refresh_prompt_ui_from_stage()            ← 不等 await！立即执行！
                                                ← 此时 _current_stage_index 已指向 Stage 3
                                                ← 所以显示的是 Stage 3 的标题和目标
   
3. 玩家看到：
   → 屏幕上方："第三试炼 · 体育馆的哨声"
   → 右侧目标："□ 走进体育馆" / "□ 面对体育老师"
   → 然后——对话框弹出来——"教室里还有课桌...学生们去哪了？"
   
   这是 Stage 2 的台词，在 Stage 3 的画面上。
```

#### 修改 B：对话触发从硬编码 match 改为数据驱动（`game_manager.gd`）

**位置：`scripts/game_manager.gd`，替换第 878-898 行**

将 `_trigger_stage_dialogue(stage: int)` 从硬编码 `match` 改为读取 MissionStage 的 `dialogue_messages` 字段：

```gdscript
## 弹出 Stage 完成时的过渡对话
## 对话数据来自 MissionStage.dialogue_messages，不再硬编码
func _trigger_stage_dialogue(cleared_stage_id: int) -> void:
    var dlg: Control = $"../HUDLayer/DialoguePanel"
    if dlg == null or not dlg.has_method("show_dialogue"):
        return
    
    var msgs: Array[Dictionary] = _get_stage_dialogue_messages(cleared_stage_id)
    if msgs.is_empty():
        return
    
    get_tree().paused = true
    dlg.show_dialogue(msgs)
    await dlg.dialogue_finished
    get_tree().paused = false


## 从 MissionManager 的链数据中按 stage_id 读取对话消息
func _get_stage_dialogue_messages(stage_id: int) -> Array[Dictionary]:
    if _mission_manager == null:
        return []
    if not _mission_manager.has_method("get_stage_dialogue"):
        return []
    return _mission_manager.get_stage_dialogue(stage_id)
```

#### 修改 C：在 MissionManager 中添加查询方法（`mission_manager.gd`）

**位置：`scripts/mission_manager.gd`，在第 77 行 `get_current_stage()` 方法之后**

```gdscript
## 按 stage_id 查找 MissionStage 的对话消息
## 用于 GameManager._trigger_stage_dialogue 数据驱动
func get_stage_dialogue(stage_id: int) -> Array[Dictionary]:
    if _chain == null:
        return []
    for stage in _chain.stages:
        if stage.stage_id == stage_id:
            return stage.dialogue_messages
    return []
```

#### 修改 D：修复时序竞态 —— `_on_stage_cleared` 必须 await 对话完成

**位置：`scripts/game_manager.gd`，第 844-856 行**

当前代码：

```gdscript
func _on_stage_cleared(cleared_stage_id: int) -> void:
    CombatFeedback.screen_shake(8.0); CombatFeedback.big_hit_stop()
    if _mission_prompt_ui and is_instance_valid(_mission_prompt_ui):
        _mission_prompt_ui.hide_direction_arrow()
    _trigger_stage_dialogue(cleared_stage_id)        # ← 不 await！
    if _mission_prompt_ui and is_instance_valid(_mission_prompt_ui):
        _refresh_prompt_ui_from_stage()              # ← 立刻执行，读到新 Stage 的数据
```

修改为：

```gdscript
func _on_stage_cleared(cleared_stage_id: int) -> void:
    CombatFeedback.screen_shake(8.0)
    CombatFeedback.big_hit_stop()
    
    # 隐藏当前方向箭头
    if _mission_prompt_ui and is_instance_valid(_mission_prompt_ui):
        _mission_prompt_ui.hide_direction_arrow()
    
    # 先完成对话，再刷新下一 Stage 的 UI
    # —— 这样对话内容和 UI 内容永远属于同一个时刻
    if cleared_stage_id < 3:  # Stage 3 不用弹对话，走胜利演出
        await _trigger_stage_dialogue(cleared_stage_id)
    
    # 对话完成后（或跳过对话后），刷新 UI
    if _mission_prompt_ui and is_instance_valid(_mission_prompt_ui):
        _refresh_prompt_ui_from_stage()
```

**注：** Stage 3（Boss Stage）完成时不走 `_trigger_stage_dialogue`——它的完结演出由 `_boss_defeated_sequence()` 接管。`_on_stage_cleared(3)` 只需要刷新 UI。

## 第三章：胜利演出时间线

### 3.1 当前实现与期望之间的差距

当前的 `_boss_defeated_sequence()`（`game_manager.gd:179-211`）：

```
当前实际流程:
  Boss HP=0 → _on_boss_killed() → _boss_defeated_sequence()
  → Boss HP Bar exit()
  → await 0.5s
  → VictoryTextUI 创建（自动播放自己的入场+停留+退场，共 2.9s）
  → await 1.0s
  → 胜利独白 Label 创建（与 VictoryTextUI 并行显示，层级混乱）
  → Vignette 设为 BOSS_DEFEAT
  → _is_post_boss_upgrade = true
  → boss_defeated signal emitted
```

**缺失的：**
1. Boss 身体坍塌动画——Boss 的 `_die()` 是普通敌人的缩放淡出，没有"坍塌"
2. 金色粒子爆发——Boss 死亡没有专属粒子效果
3. 学生小怪消散——如果有残余的"学生"敌人在场上，它们应该消散
4. VictoryTextUI 的等待时间不够——代码只 `await 1.0s`，但 VictoryTextUI 的停留时间有 2.0s，导致独白文字覆盖在胜利文字上
5. 升级面板弹出过早——`_on_level_up_available` 在 `_boss_fight_active=true` 时会累积 `_pending_boss_upgrades` 但不弹出，但 `_on_boss_defeated()` 把 `_boss_fight_active` 设为 `false` 后，累积的升级没有立即处理

### 3.2 目标时间线

           - 冻结 Boss 的 AI 移动（不再追击）

           - CombatFeedback.big_hit_stop() — 6帧（0.1s）顿帧

           - 0.50s: 退场 — modulate.a 1→0 + 向上浮动 20px

## 尾声：为什么这次重构值得认真做

在这个项目里，已经有太多人把精力花在了"让东西能动起来"的层面。战斗系统的框架、敌人的 AI 行为、技能的组合——这些都是"对"的事情。但如果一个玩家打完 Stage 2、走进体育馆的时候，屏幕弹出的是一句五分钟前就该听到的台词——那么前面所有的"对"都等于白做。

因为玩家感受到的不是代码逻辑。玩家感受到的是**时间线**。

一个好的时间线，让玩家觉得"我在一个故事里"。一个错位的时间线，让玩家觉得"我在一个程序里"。

这次的修复看起来只是改了几个 `await` 的位置、把几行硬编码的对话挪到了数据文件里。但它们做的事情是一样的：**把说每一句话的时刻，还给它应该属于的那一帧。**

佐藤的哨声不能在第一波小怪打完的时候响。
"下课"两个字不能出现在玩家还在找校门的路上。

每个人、每句台词、每个粒子——都有它该去的那一帧。

我们的工作不是制造它们。是护送它们到达。

*—— 小岛秀夫*
