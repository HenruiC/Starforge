### 0.2 根因总结

`map_system.gd` 是为旧的 3200×2400 地图编写的。当地图扩展到 5760×4320 时（`school_map.gd` 已完成扩展），地图系统完全未同步更新。`_build_map_viewport()` 是一个"快速修复尝试"，但它用海量 ColorRect 节点绕过了 Image-based 渲染的正确方向。

**一句话：两个系统活在两个不同的地图里。**

## 一、选定技术方案

### 1.1 方案对比

| 方案 | 原理 | 节点数 | 迷雾更新 | 性能 | 复杂度 |
|------|------|:---:|------|:---:|:---:|
| **A: Image纹理+动态Fog（选定）** | 两张 ImageTexture 叠加：底图(Map) + 上层(Fog)，迷雾通过修改 Fog Image 像素实现 | 2 个 TextureRect | 修改 120×90 像素 Image，毫秒级 | 最优 | 低 |
| B: ColorRect网格 | 每个 tile 一个 ColorRect，逐格控制颜色和透明度 | 10,800 个 Control 节点 | 遍历修改 modulate | 灾难 | 低 |
| C: SubViewport+Camera2D | 第二相机俯拍关卡，输出到 ViewportTexture | 3-4 个节点 | 自动（所见即所得） | 中等 | 高 |
| D: TileMapLayer+Shader | 用第二个 TileMapLayer 配合 shader 做迷雾 | 2 个节点 | Shader 计算 | 最优 | 极高 |

**选定方案 A 的理由：**

1. **与现有代码方向一致** — `map_system.gd` 的 `_draw_fog()` 已经在用 Image.create() + ImageTexture 方案，只是尺寸错误。这是修复，不是重写。
2. **节点数极低** — 2 个 TextureRect（MapTexture + FogOverlay）+ 若干 Label（标记），总共不超过 10 个节点。
3. **迷雾更新零开销** — 120×90=10,800 像素的 Image 操作在 Godot 中耗时 <1ms。仅在玩家发现新 tile 时才重建 texture。
4. **分离关注点** — 底图纹理（`school_map.gd::generate_map_texture()`）和迷雾纹理（`MapSystem`）完全独立，可并行开发。
5. **100% 可控** — 每个像素的颜色/透明度由 GDScript 精确控制，不依赖 shader 或渲染管线。

方案 B 已在实际代码中实施（`_build_map_viewport()`），结果证实了之前技术讨论的预言：**10,800 个节点是灾难**。方案 C 对当前需求过度设计——我们不需要"实时同步关卡变化"的地图，我们需要的是"探索记忆"式地图。方案 D 引入了 Godot 4.x 的 shader 复杂性，团队当前没有 shader 程序员。

**设计决策：底图不完全隐藏，迷雾层叠上**

宫崎英高式做法：即使未探索区域，玩家也能看到一个"地形轮廓暗示"——底图以极低透明度显示，迷雾覆盖其上。这让玩家隐约感知"还有更大的空间"，制造期待和不安。

小岛秀夫式调整：实际上我们让底图**完全可见**，但覆盖半透明迷雾。已探索区域迷雾移除，露出底图全色。这样玩家看到"我去过的地方"是彩色的，"没去过的地方"是黑暗中的轮廓——就像记忆。

**实施简化**：底图始终可见，迷雾 alpha 决定可视程度。这是最直接的实现方式。

## 二、数据结构设计

### 2.1 核心常量

```gdscript
# MapSystem 新常量（与 school_map.gd 保持一致）
const MAP_WIDTH_TILES := 120     # 5760 / 48
const MAP_HEIGHT_TILES := 90     # 4320 / 48
const TILE_SIZE := 48            # 每 tile 像素数
const MAP_WIDTH_PX := 5760       # 世界像素宽度
const MAP_HEIGHT_PX := 4320      # 世界像素高度

# 显示常量
const MAP_DISPLAY_SCALE := 6     # 每 tile 放大到 6px（120×6=720, 90×6=540）
const MAP_DISPLAY_W := 720       # 地图显示宽度
const MAP_DISPLAY_H := 540       # 地图显示高度

# 探索常量
const EXPLORE_RADIUS_TILES := 5  # 玩家周围探索半径（tile 单位）
const EXPLORE_RADIUS_SQ := 25    # 半径平方（5²），用于圆形探索范围
```

### 2.2 迷雾网格

```gdscript
## 迷雾状态网格 [x][y] → bool
## true  = 已探索（迷雾消散）
## false = 未探索（迷雾覆盖）
## 维度: MAP_WIDTH_TILES × MAP_HEIGHT_TILES
var _fog_grid: Array[Array] = []  # Array[Array[bool]]

## 已探索 tile 计数（用于探索率计算）
var _fog_explored: int = 0

## 总 tile 数 = MAP_WIDTH_TILES × MAP_HEIGHT_TILES = 10,800
var _fog_total: int = 10800
```

**初始化**：`_fog_grid` 是一个 `Array[Array]`，外层 120 个元素（x 方向），内层每个包含 90 个 `bool`（y 方向）。全部初始化为 `false`。

**为什么不是 PackedByteArray？**
- `Array[Array]` 的随机访问 `_fog_grid[x][y]` 在 GDScript 中足够快（10,800 个元素）
- 探索更新频率极低（每秒最多几次），不需要连续内存优化
- GDScript 的 `Array[Array]` 更易读写和调试
- 如果性能成为瓶颈（不可能），可以直接替换为 `PackedByteArray`，接口不变

### 2.3 标记数据结构

```gdscript
## 地图标记类型
enum MarkerType {
    PLAYER,       # 玩家位置 — 红色圆点
    BOSS,         # Boss 位置 — 暗红色菱形
    QUEST_TARGET, # 任务目标 — 金色旋转星
    ZONE_EXIT,    # 区域出口 — 绿色箭头
}

## 单个标记数据
class MapMarker:
    var type: MarkerType
    var world_position: Vector2     # 世界坐标（像素）
    var label_text: String          # 标签文字（可选）
    var visible: bool = true        # 是否显示
    var pulsate: bool = false       # 是否脉动动画

## 标记列表
var _markers: Array[MapMarker] = []
```

### 2.4 地图纹理缓存

```gdscript
## 底图纹理（由 school_map.gd 生成，120×90 像素）
var _base_map_texture: ImageTexture = null

## 迷雾纹理（动态生成，120×90 像素）
var _fog_texture: ImageTexture = null

## 迷雾 Image 对象（用于修改像素后重建纹理）
var _fog_image: Image = null

## 是否需要重建迷雾纹理（脏标记模式）
var _fog_dirty: bool = false
```

## 四、交互规范

### 4.1 地图打开/关闭

```
状态转换：
  PLAYING ──[M 键]──→ MAP (暂停游戏)
  MAP     ──[ESC/M]─→ PLAYING (恢复游戏，Engine.time_scale=1.0)

入场动画 (0.25s):
  t=0.00   MapPanel.visible = true
  t=0.00   设置 GameState.State.MAP（自动暂停）
  t=0.00   MapPanel/Bg.color.a: 0 → 0.85 (0.15s, EASE_OUT)
  t=0.05   MapPanel/Title.modulate.a: 0 → 1 (0.15s)
  t=0.05   MapPanel/MapTexture.modulate.a: 0 → 1 (0.2s)  
  t=0.10   MapPanel/FogOverlay: 立即显示（无动画，迷雾不需要渐显）
  t=0.10   玩家标记/任务标记/FogOverlay: 淡入 (0.1s)
  t=0.15   ZoneLabel/Hint: 淡入 (0.1s)
  t=0.25   动画结束，地图完全可用

退场动画 (0.15s):
  t=0.00   全部淡出 (0.15s)
  t=0.15   MapPanel.visible = false
  t=0.15   恢复 GameState.State.PLAYING（引擎自动取消暂停）
```

**设计理由（小岛秀夫）**：入场 0.25s 而不是瞬间显示——这给玩家一个"切换思维"的瞬间。地图不是 HUD，地图是一个仪式。你停下来，打开它，环顾四周，然后关上它继续前进。就像《死亡搁浅》中 Sam 查看地图的动作。

### 4.2 迷雾消散

```
机制：
  迷雾在游戏进行中持续更新（不在暂停时更新，因为地图打开=暂停）

  更新频率：每 0.25s (4fps @ 60fps) — 不是每帧
  更新触发：玩家 tile 坐标变化时

  消散方式：
  1. 获取玩家当前 tile 坐标 (tx, ty)
  2. 以 (tx, ty) 为圆心，EXPLORE_RADIUS_TILES (5) 为半径
  3. 遍历圆形区域内的所有 tile
  4. 如果 _fog_grid[nx][ny] == false：
     a. 设为 true
     b. _fog_explored += 1
     c. 设置 _fog_dirty = true
  5. 如果 _fog_dirty：
     a. 修改 _fog_image 对应像素: set_pixel(nx, ny, Color(0,0,0,0))
     b. _fog_texture = ImageTexture.create_from_image(_fog_image)
     c. FogOverlay.texture = _fog_texture
     d. _fog_dirty = false

  视觉风格 — 宫崎英高式：
  迷雾的消散是离散的（按 tile，不是平滑渐变）。这是有意的——
  每一个 tile 被揭开时，玩家看到的是"一块记忆"，而不是"一个模糊地带"。
  就像黑暗之魂中的地图碎片——你不是在逐渐看清，你是在收集世界。
```

### 4.3 地图打开时的迷雾刷新

地图打开时（MAP 状态），游戏已暂停，但迷雾数据应该反映最新的探索状况：

```
_toggle_map() → 显示 MapPanel 前:
  1. 调用 MapSystem.update() —— 确保迷雾与当前位置同步
  2. 设置 FogOverlay.texture 为最新迷雾纹理
  3. 更新播放器标记位置
  4. 显示 MapPanel
```

### 4.4 玩家标记位置计算

```
世界坐标 → 地图显示坐标的映射：

  tile_x = player.global_position.x / TILE_SIZE       # 0..119
  tile_y = player.global_position.y / TILE_SIZE       # 0..89

  地图显示区域（相对于 MapTexture 节点）:
    display_x = tile_x * MAP_DISPLAY_SCALE             # 0..720
    display_y = tile_y * MAP_DISPLAY_SCALE             # 0..540

  MapTexture 的 anchor 在 (0.5, 0.5)，offset 是 (-360, -290)
  所以 PlayerMarker 的 offset:
    offset_left   = -360 + display_x - marker_width/2
    offset_right  = -360 + display_x + marker_width/2
    offset_top    = -290 + display_y - marker_height/2
    offset_bottom = -290 + display_y + marker_height/2

  简化（使用 Label 的 anchor 居中模式）:
    将 PlayerMarker 的 anchor_left/anchor_right 设为与 MapTexture 相同
    offset_left  = -360 + display_x - 8
    offset_right = -360 + display_x + 8
    offset_top   = -290 + display_y - 8
    offset_bottom = -290 + display_y + 8
```

## 八、风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|:---:|------|------|
| `Image.create_from_image()` 在 Godot 4.6 中行为变化 | 低 | 高 | 当前代码已调用此 API（`_draw_fog()`），仅尺寸变化；如果出错，fallback 用 `ImageTexture.new()` + `update()` |
| 120×90 纹理拉伸到 720×540 后颜色混合模糊 | 低 | 中 | 底图是纯色块，无渐变——即使最近邻插值也清晰。如需锐利，TextureRect 设置 `texture_filter = TEXTURE_FILTER_NEAREST` |
| `Array[Array]` 在大型地图上性能退化 | 极低 | 低 | 10,800 元素远低于 GDScript 数组性能阈值。如果未来地图再扩大 10 倍，可迁移到 PackedByteArray |
| MapSystem 在 MAP 状态下被暂停（INHERIT 导致 update() 不执行） | 中 | 中 | MAP 状态暂停后迷雾不更新是预期行为（游戏已暂停）。地图打开前调用一次 update() 即可。MapSystem 不需要 _process |
| 与旧地图相关的 zone_definitions 坐标不一致 | 低 | 高 | 地图扩展方案文档已更新 zone 坐标（`地图扩展方案-关卡.md` 3.10 节），与本文档独立 |

## 九、未来扩展（本期不做，预留接口）

### 9.1 平滑迷雾消散（像素级渐变）

当前实现是 binary（探索/未探索）。未来可以改为：每个 tile 存储 0..255 的探索程度，`_fog_image` 使用 alpha 渐变。但这需要 `Image.FORMAT_RGBA8` 的每个像素 alpha 从 255 递减到 0，视觉效果更流畅。

**预留接口**：`_fog_grid` 从 `Array[Array[bool]]` 改为 `Array[Array[int]]`（0-255），`_draw_fog()` 中使用 `Color(0.02, 0.02, 0.06, float(255 - value) / 255.0)`。

### 9.2 地图缩放/拖动

当支持缩放/拖动时：
- MapTexture 和 FogOverlay 放入一个 `ScrollContainer` 或使用 `Camera2D`（新的 SubViewport）
- 缩放级别：1x/2x/4x（tile 从 6px→12px→24px）
- 拖动：鼠标拖拽平移，边界限制在地图范围内

**预留接口**：`MapSystem` 提供 `world_to_map_pos()` 和 `map_to_world_pos()` 静态方法，供缩放/拖动系统调用。

### 9.3 小地图

屏幕角落的圆形迷你地图，显示玩家周围 20×15 tiles 的区域：
- 使用同一个 `_fog_grid` 数据源
- 新增 `MiniMap` 节点（TextureRect，100×75 像素，圆形遮罩）
- 每帧更新：裁剪玩家周围区域 → 创建小 Image → 渲染

**预留接口**：`MapSystem.get_fog_grid()` 返回 `_fog_grid` 引用，`MiniMap` 只读不写。

### 9.4 地图碎片收集

黑暗之魂/艾尔登法环式的地图收集机制：
- 初始：玩家没有任何地图——打开只看到黑色背景 + "你还没有地图。"
- Stage 1 完成后：获得"学校周边地区手绘地图"（显示操场和教学楼轮廓）
- Stage 2 完成后：获得"教室分布图"（教学楼内部细化）
- Stage 3 完成后：获得"体育馆结构图"（Boss间详细布局）

**预留接口**：`MapSystem` 的 `_base_map_texture` 可以按阶段逐步替换为更详细的纹理。`MapData` 类包含 `detail_level: int`（0-3）。

## 十、设计决策记录

### Q1: 为什么底图始终显示而不是完全隐藏未探索区域？

**宫崎英高**：在《艾尔登法环》中，未获得地图碎片的区域显示为空白羊皮纸——你知道"这里有什么"，但不知道细节。完全隐藏会让玩家觉得"地图坏了"而不是"还没探索"。

**小岛秀夫**：我同意。这就像《死亡搁浅》中 Sam 的 BB 探测器——你会看到地形轮廓，但细节需要你亲自走过。完全黑暗 = 没有信息 = 玩家不信任地图。半透明迷雾 = 被遮挡的信息 = 玩家被激励去揭示。

### Q2: 为什么迷雾消散是离散 tile 而不是平滑像素级？

**宫崎英高**：离散 = 你能清楚地区分"我去过"和"我没去过"。这是探索的反馈。平滑渐变的危险是玩家无法准确判断探索边界。

**小岛秀夫**：技术上更简单。设计上更强。We don't need to do what everyone else is doing。

### Q3: 为什么不在 MAP 状态下持续更新迷雾？

**一致决策**：MAP 状态 = 游戏暂停。暂停状态下的任何更新都会导致"玩家看地图时迷雾在动"的诡异体验。正确的做法是：地图打开前确保数据已同步，显示的是"此刻的记忆"。

### Q4: _process 中每 0.25s 更新迷雾会不会太慢？

**卡马克**：每帧更新 120×90 Image 是不必要的。玩家移动速度约 200 px/s = 约 4 tiles/s。0.25s 的更新间隔意味着玩家最多移动 1 tile 才更新一次——视觉上完全不可察觉。而且只在 `_fog_dirty` 为 true 时才重建纹理，绝大多数帧不做任何 Image 操作。
