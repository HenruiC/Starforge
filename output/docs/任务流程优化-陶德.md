# 任务流程优化方案 — 陶德·霍华德

> "See that door? You should be able to lock it. Then the player has to figure out how to get through. That's not a bug — that's a story."

---

## 一、 现有系统诊断

### 1.1 已拥有的强大基础

这套系统比我想象的成熟得多：

- **6 种 Trigger 类型**已经定义好了（`TriggerConfig.MissionTriggerType`），其中 `TIME_SURVIVE` 的计时器逻辑已经实现在 `MissionTriggerManager._process()` 里
- **Zone 系统**已经完整运行：动态创建、场景扫描、body_entered/exited 信号转发
- **Stage 激活条件**（`activation_condition`）已经支持延迟激活，`_check_pending_activation()` 框架就绪
- **`EventBus`** 已经是全局事件枢纽，`objective_completed` 信号已经有了
- **所有配置数据是 Resource** — `MissionChain` / `MissionStage` / `MissionObjective` / `TriggerConfig` 全部继承自 Resource，天然支持 `.tres` 文件序列化

### 1.2 零代码承诺：新副本 = 纯数据配置

**这是本次优化的核心约束。** 我把它写在最前面，因为它决定所有设计决策：

> 当设计师说"我要做一个工厂副本"的时候，他应该只做一件事：**写一个 `.tres` 文件。**

具体来说：
- **DEFEND_ZONE / PROTECT_OBJECT 做成通用 Trigger 类型** — 与"学校"无关，与任何具体副本无关。它们是引擎级别的任务原语。
- **门系统做成可配置节点** — `LockedDoor` 接受任意 `door_id` + 任意 `door_tile_positions`。不硬编码"体育馆入口"的坐标。
- **新副本只需配置链数据** — 程序代码（`MissionTriggerManager`、`_process` 驱动、Evaluator 注册）写一次。副本差异完全由 `MissionChain` 资源的参数组合体现。
- **Stage / Objective / Trigger 三层分离保持不变** — 这是现有架构的基石，新增 Trigger 类型只是给引擎增加动词，不改变语法。

### 1.3 当前的问题

但玩家会怎么想？玩家玩完四关之后会说：

> *"我做的事情就是 — 走到一个地方，杀一些东西，走到下一个地方，杀更多东西。然后打 Boss。"*

这不是一个 Bethesda 游戏的问题。这是一个 **玩家没有机会说"你猜怎么着"** 的问题。

**缺少的东西：**
1. **变奏** — 每一关都是一个节拍，没有切分音
2. **时间压力** — 没有"我必须撑住"的时刻
3. **目标多样性** — 杀敌是唯一对世界产生反馈的行为
4. **环境互动** — 门是装饰，区域没有"守"的意义

## 二、 新 Trigger 类型设计

我决定新增 **2 种 Trigger 类型** + **1 种复合机制**。DEFEND_ZONE 和 PROTECT_OBJECT 是新类型；KILL_TO_UNLOCK 通过扩展现有 KILL_COUNT + 新增 Objective 完成回调实现，不需要新 Trigger。

### 2.1 DEFEND_ZONE（Trigger Type 7）— 通用 Trigger

> **通用性：适用于任何副本中"要求玩家守住某个位置"的场景。**
> 学校走廊、工厂控制室、太空站气闸——都用同一个 Trigger 类型，只换 `zone_id` 和 `target_value`。

**核心理念：**
> 玩家进入一个区域后，必须留在里面坚持 X 秒。出去就暂停计时——"你想跑？计时器等着你回来。"

**设计细节：**

| 维度 | 内容 |
|------|------|
| **触发条件** | 玩家进入参数中指定的 `zone_id` 的 Area2D |
| **完成条件** | 在区域内累计存活 `target_value` 秒 |
| **暂停机制** | 玩家离开区域时计时器暂停（不重置），重新进入后继续 |
| **重置模式（可选）** | `reset_on_exit=true` 时离开区域计时归零（更严苛的变体） |
| **失败条件** | 玩家死亡（已由 `player_died` → chain failed 覆盖） |
| **UI 提示** | 计时器进度条 + 描述文字由 Objective 的 `description` 字段驱动 |
| **敌人行为** | 由副本设计师在关卡中预置，或通过 `completion_action` 触发波次——Trigger 本身不关心敌人从哪来 |
| **params 定义** | `{"zone_id": "<任意 zone_id>", "pause_on_exit": true, "reset_on_exit": false}` |

**跨副本使用示例：**

```
// 学校副本
TriggerConfig { trigger_type=7, target_value=30.0,
  params={"zone_id": "zone_hallway_defend", "pause_on_exit": true} }

// 工厂副本
TriggerConfig { trigger_type=7, target_value=45.0,
  params={"zone_id": "zone_control_room", "pause_on_exit": true} }

// 太空站副本（严苛模式 — 离开一秒就重来）
TriggerConfig { trigger_type=7, target_value=20.0,
  params={"zone_id": "zone_airlock", "pause_on_exit": true, "reset_on_exit": true} }
```

**实现路径：**

1. `TriggerConfig.MissionTriggerType` enum 新增 `DEFEND_ZONE = 7`
2. `MissionTriggerManager._register_evaluators()` 注册 `_eval_defend_zone`（一次注册，永久生效）
3. `_eval_defend_zone()` 逻辑：
   - 在 `_process()` 中检查：当前 Stage 是否有 DEFEND_ZONE 类型的 Objective
   - 从 `params` 中读取 `zone_id`（不硬编码任何 zone 名）
   - 如果 `pause_on_exit == true`：检查玩家是否在 zone 内（通过 `_tracked_zone_areas` 的 overlapping_bodies）
   - 玩家在 zone 内 → `current_value += delta`；不在 → 计时暂停
   - 如果 `reset_on_exit == true`：离开时 `current_value = 0`
4. 新 EventBus 信号：`defend_zone_state_changed(zone_id: String, is_active: bool)` — 供 UI / 其他系统响应

**为什么不是 TIME_SURVIVE 的扩展？**

TIME_SURVIVE 是纯计时器——只要你还活着，时间就在走。DEFEND_ZONE 多了一个空间锚点。它们本质不同：TIME_SURVIVE 考验的是"你能活多久"，DEFEND_ZONE 考验的是"你敢不敢留在这里"。把 zone 参数塞进 TIME_SURVIVE 会让 `params` 语义混乱。两个独立类型让副本设计师的意图更清晰。

### 2.2 PROTECT_OBJECT（Trigger Type 8）— 通用 Trigger

> **通用性：适用于任何副本中"保护场景物体不被摧毁"的场景。**
> 发电机、魔法阵、NPC、数据终端——都是同一个 Trigger 类型，只换 `object_id`。

**核心理念：**
> 场景里有一个东西——可能是魔法核心、可能是被困的 NPC——你必须保护它不被破坏。敌人会主动攻击它。

**设计细节：**

| 维度 | 内容 |
|------|------|
| **触发条件** | Stage 激活后自动开始追踪 |
| **完成条件** | 保护目标 HP > `target_hp_threshold` 且累计存活 `target_value` 秒 |
| **失败子条件** | 保护目标 HP <= `target_hp_threshold` → Objective 标记失败 |
| **UI 提示** | 目标血条 + 计时器 + 描述文字由 Objective 的 `description` 字段驱动 |
| **敌人行为** | `ProtectableObject` 节点自带 `attracts_enemy_attention: bool` 开关——启用时敌人 AI 优先攻击它 |
| **params 定义** | `{"object_id": "<任意 object_id>", "target_hp_threshold": 0}` |

**跨副本使用示例：**

```
// 学校副本（保护魔法阵）
TriggerConfig { trigger_type=8, target_value=30.0,
  params={"object_id": "protect_sealing_circle", "target_hp_threshold": 0} }

// 工厂副本（保护发电机 — HP 掉到 50% 以下直接失败）
TriggerConfig { trigger_type=8, target_value=45.0,
  params={"object_id": "generator_main", "target_hp_threshold": 0.5} }

// 太空站副本（保护逃生舱 — 可以被打但不能爆）
TriggerConfig { trigger_type=8, target_value=60.0,
  params={"object_id": "escape_pod", "target_hp_threshold": 0} }
```

**`target_hp_threshold` 设计：**
- `0` = 目标 HP 归零则失败（最常用）
- `0.5` = 目标 HP 低于 50% 则失败（更严格——"不能让它受伤"）
- `-1`（特殊值）= 目标可以被破坏但 Objective 仍可完成（剧情向——"它注定会毁掉，但你要坚持够久"）

**实现路径：**

1. `TriggerConfig.MissionTriggerType` enum 新增 `PROTECT_OBJECT = 8`
2. 新脚本 `scripts/protectable_object.gd`：
   ```
   class_name ProtectableObject
   extends StaticBody2D
   - max_health, _health
   - take_damage(amount)
   - 死亡时 emit EventBus.protectable_destroyed(object_id)
   - HP bar 显示
   - 视觉：ColorRect + 闪烁受伤反馈
   ```
3. `MissionTriggerManager._register_evaluators()` 注册 `_eval_protect_object`
4. `_eval_protect_object()` 逻辑：
   - 在 `_process()` 中查找 `object_id` 对应的 ProtectableObject 节点
   - 如果 ProtectableObject 存在且 HP > threshold → `current_value += delta`
   - 如果 ProtectableObject 被销毁 → Objective 失败，emit 特殊事件
   - 需要监听 `EventBus.protectable_destroyed`
5. 新增 EventBus 信号：
   - `protectable_destroyed(object_id: String)` — 保护目标被摧毁
   - `protectable_damaged(object_id: String, current_hp: int, max_hp: int)` — 受伤反馈

**ProtectableObject 脚本设计（纯配置驱动）：**

```gdscript
class_name ProtectableObject
extends StaticBody2D

## 可被保护的目标物体 — 纯配置驱动，不绑定任何副本
## 场景中实例化，填 export 参数即可。由 MissionTriggerManager 追踪其存活状态。

## 唯一标识 — 与 TriggerConfig.params.object_id 匹配
@export var object_id: String = ""

## 最大 HP
@export var max_health: int = 100

## 是否吸引敌人优先攻击（启用后敌人 AI 在范围内时优先攻击此物体）
@export var attracts_enemy_attention: bool = true

## 受伤时是否闪烁
@export var flash_on_damage: bool = true

@onready var sprite: ColorRect = $Sprite
@onready var hp_bar: ProgressBar = $HealthBar
@onready var collision: CollisionShape2D = $CollisionShape2D

var _health: int

func _ready() -> void:
    _health = max_health
    add_to_group("protectable_object")
    hp_bar.max_value = max_health
    hp_bar.value = _health
    hp_bar.visible = false

## 受到伤害（由敌人 AI 的接触伤害 / 远程子弹调用）
func take_damage(amount: int) -> void:
    if _health <= 0:
        return
    _health = max(_health - amount, 0)
    hp_bar.value = _health
    hp_bar.visible = true

    EventBus.protectable_damaged.emit(object_id, _health, max_health)

    # 受伤闪烁反馈
    if flash_on_damage and sprite:
        sprite.modulate = Color.WHITE
        var t := create_tween()
        t.set_process_mode(Tween.TWEEN_PROCESS_IDLE)
        t.tween_property(sprite, "modulate", Color(1.0, 0.3, 0.3), 0.1)
        t.tween_property(sprite, "modulate", Color.WHITE, 0.2)

    # HP 归零
    if _health <= 0:
        EventBus.protectable_destroyed.emit(object_id)
        # 销毁动画
        var t_die := create_tween()
        t_die.set_process_mode(Tween.TWEEN_PROCESS_IDLE)
        t_die.tween_property(self, "modulate:a", 0.0, 0.5)
        t_die.tween_callback(queue_free)

## 查询：此物体是否存活（供 MissionTriggerManager 的 _eval_protect_object 调用）
func is_alive() -> bool:
    return _health > 0

## 查询：HP 百分比
func hp_percent() -> float:
    return float(_health) / float(max_health)
```

**配置方式：场景编辑器中拖一个 ProtectableObject 节点到关卡中，填 `object_id` 和 `max_health`。** `MissionTriggerManager` 运行时通过 `get_tree().get_nodes_in_group("protectable_object")` 查找匹配的 `object_id`。

**敌人 AI 修改：**
- `Enemy` 需要新增 `prefer_target_object: bool` 和 `target_object_id: String`
- 当 `prefer_target_object == true` 且 ProtectableObject 在攻击范围内时，优先攻击目标
- 这个字段的默认值是 `false`——只有副本设计师在预置敌人配置中显式设置时才会激活
- 在 `get_preplaced_enemies()` 的配置 dict 中增加 `"target_object": "generator_main"` 即可

### 2.3 KILL_TO_UNLOCK（复合机制，非新 Trigger 类型）— 通用 completion_action

> **通用性：`completion_action` 是在 Objective 层定义的世界反馈协议。**
> `unlock_door` 只是第一个 action 类型。未来任何 Objective 完成都可以触发任意世界变化——不限于门，也不限于学校副本。

**核心理念：**
> 这不是新 Trigger。这是"KILL_COUNT + `completion_action`"的复合。击败指定的精英敌人 → 发出 `door_unlock_requested` 信号 → LockedDoor 节点响应并开门。杀人是手段，开门是结果。**Trigger 只关心"杀没杀"；`completion_action` 关心"杀了之后世界怎么变"——两者解耦。**

**设计决策——为什么不做新 Trigger 类型：**

KILL_TO_UNLOCK 的"击杀"部分完全被现有 KILL_COUNT + `enemy_filter="elite"` 覆盖。唯一的增量是"击败后开门"。如果我为这个组合创建一个新 Trigger 类型，那未来每一对"做A → 改变世界B"的组合都需要新类型。这是死路。

正确做法是：**给 Objective 增加 `completion_action: Dictionary` 概念**——一个 Objective 完成后，系统检查它的 `completion_action`，并向 EventBus 发送对应的信号。LockedDoor 只监听信号，不关心是哪个 Objective 触发的。

**`completion_action` 协议设计：**

```
completion_action = {"type": "unlock_door", "door_id": "<任意 door_id>"}
```

- `type` 决定分发到哪个 EventBus 信号
- 其余字段是信号的 payload
- 副本设计师在 .tres 中配置，零代码

**设计细节：**

| 维度 | 内容 |
|------|------|
| **Objective 的 Trigger** | KILL_COUNT，`enemy_filter="elite"`，`target_value=1`（或其他任意组合） |
| **`completion_action`** | `{"type": "unlock_door", "door_id": "<副本设计师指定的门ID>"}` |
| **门锁响应** | LockedDoor 监听 `EventBus.door_unlock_requested(door_id)` → 对照自己的 `door_id` → 播放开门动画 |
| **地图反馈** | 锁着的门 tile（特殊颜色 source 10）替换为地板 tile |

**跨副本使用示例：**

```
// 学校副本 — 守卫精英开门
MissionObjective {
  trigger = KILL_COUNT { enemy_filter="elite", target=1 }
  completion_action = {"type": "unlock_door", "door_id": "gym_lock_door"}
}

// 工厂副本 — 击败工头开办公室门
MissionObjective {
  trigger = KILL_COUNT { enemy_filter="boss", target=1 }
  completion_action = {"type": "unlock_door", "door_id": "office_door"}
}

// 太空站副本 — 击败三个守卫才能开门（同一个 Objective 配 target=3）
MissionObjective {
  trigger = KILL_COUNT { enemy_filter="elite", target=3 }
  completion_action = {"type": "unlock_door", "door_id": "bridge_door"}
}
```

**实现路径：**

1. `MissionObjective` 新增字段：
   ```
   @export var completion_action: Dictionary = {}
   // {"type": "unlock_door", "door_id": "xxx"}
   // 未来可扩展: {"type": "spawn_enemies", "count": 5}, {"type": "activate_zone"}
   ```
2. `MissionTriggerManager._on_objective_completed()` 中增加：
   ```
   if not obj.completion_action.is_empty():
       _dispatch_completion_action(obj)
   ```
3. `_dispatch_completion_action(obj)` 根据 `type` 分发：
   - `"unlock_door"` → `EventBus.door_unlock_requested.emit(obj.completion_action["door_id"])`
4. 新 EventBus 信号：`door_unlock_requested(door_id: String)`
5. 新脚本 `scripts/locked_door.gd` 或扩展 `scripts/door.gd`：
   ```
   监听 EventBus.door_unlock_requested
   收到匹配的 door_id → _open() + 替换 TileMap tile
   ```

## 三、 学校副本新任务链

### 3.1 旧 vs 新 对比

| Stage | 旧版 | 新版 |
|-------|------|------|
| Stage 1 "校门" | 到达校门 + 击杀8个 | **不变**（已有节奏很好，作为教学关够用） |
| Stage 2 "铃声下的教室" | 击杀15个 → 到达体育馆入口 | **击杀15个 → 走廊驻守30秒** → 到达体育馆入口 |
| Stage 3 "体育馆的哨声" | 到达入口(条件) → Boss间 + 杀Boss | **到达入口 → 击败精英守卫 → 门开 →** Boss间 + 杀Boss |
| Stage 4 "下课铃" | 逃离校门 | **不变**（收束需要稳定，不加复杂度） |

### 3.2 Stage 2 详细设计：「铃声下的教室」

```
┌──────────────────────────────────────────────────────────┐
│  Stage 2 "铃声下的教室"                                    │
│                                                          │
│  Objective A: 击杀 15 个敌人                               │
│    trigger: KILL_COUNT, enemy_filter="any", target=15     │
│    UI: "走廊上全是——别省技能 (x/15)"                       │
│                                                          │
│  Objective B: 在走廊驻守 30 秒（Objective A 完成后激活）      │
│    trigger: DEFEND_ZONE, zone_id="zone_hallway_defend"    │
│    params: {"pause_on_exit": true}                       │
│    target_value: 30.0                                    │
│    UI: "守住走廊 (xx.x/30.0s)"                           │
│                                                          │
│  Objective C: 前往体育馆入口                               │
│    trigger: LOCATION_REACH, zone_id="zone_gym_entrance"   │
│    UI: "前往体育馆"                                       │
│                                                          │
│  完成提示: "✓ 走廊安静了"                                   │
│                                                          │
│  驻守期间 — 波次从两侧教室涌出：                             │
│    • 左侧教室（左教学楼南侧门洞） → 2-3 melee/波              │
│    • 右侧教室（右教学楼南侧门洞） → 2-3 melee/波              │
│    • 连廊方向 → 1-2 ranged/波（远程压制）                   │
│    • 约每 8 秒一波，30 秒内约 3-4 波                        │
│    • 敌人总数控制在 15-20 个（含 Objective A 的 15 个）       │
└──────────────────────────────────────────────────────────┘
```

**设计理由（陶德的回答）：**

> *"为什么是杀15个然后驻守，而不是反过来？"*
>
> 因为先杀15个让你觉得自己很强。然后驻守30秒让你发现，"等等，它们一直来？"——这是恐惧时刻。如果先驻守再杀，节奏是"熬过去→清场→无聊"。现在这个顺序，玩家到30秒最后5秒必定在喊："门在哪？！"

**Zone 定义（新增）：**

```
{
    "zone_id": "zone_hallway_defend",
    "position": Vector2(2640, 3000),    // 连廊与教学楼交汇区域
    "size": Vector2(960, 576),          // 20x12 tile
}
```

这个 zone 覆盖连廊中段 + 两侧教室门洞前的走廊。玩家不能跑出这个区域，否则计时器暂停。这强迫玩家在狭窄的连廊里"接敌"——不能风筝，不能逃跑。

**波次触发：**

Objective B（DEFEND_ZONE）激活时，通过 `completion_action` 或 Stage 激活回调触发临时波次生成。在 `GameManager` 中：
- 当检测到当前 Stage 有 DEFEND_ZONE 类型的 Objective 变为 active 时
- 启动一个临时 `defend_spawn_timer`，每 8 秒从两侧教室门洞位置生成一波敌人
- 当 DEFEND_ZONE Objective 完成（30秒到）后立即停止生成

### 3.3 Stage 3 详细设计：「体育馆的哨声」

```
┌──────────────────────────────────────────────────────────┐
│  Stage 3 "体育馆的哨声"                                    │
│                                                          │
│  activation_condition: LOCATION_REACH, zone_gym_entrance   │
│                                                          │
│  Objective A: 击败入口精英守卫                              │
│    trigger: KILL_COUNT, enemy_filter="elite", target=1    │
│    completion_action: {"type": "unlock_door",              │
│                        "door_id": "gym_lock_door"}        │
│    UI: "击败守卫 · 开门 (0/1)"                             │
│                                                          │
```

## 四、 门系统完整设计 — 完全可配置

> **核心原则：门不硬编码任何坐标。任何 TileMap + 任何 tile 位置 + 任何 door_id = 一扇锁门。**

### 4.1 LockedDoor 脚本 — 纯配置驱动

```gdscript
class_name LockedDoor
extends StaticBody2D

## 锁住的门 — 由 EventBus.door_unlock_requested(door_id) 事件解锁
## 完全可配置：不绑定任何副本、任何坐标、任何 tile 位置
##
## 用法：在任意场景中实例化一个 LockedDoor 节点，填以下 export 参数即可。

## 门的唯一 ID — 与 Objective 的 completion_action.door_id 匹配
@export var door_id: String = ""

## TileMap 引用（通过 NodePath 获取）
@export var tilemap_path: NodePath

## 锁门 tile 在 TileSet 中的 source_id（用于擦除时的查找）
@export var lock_source_id: int = 10

## 开门后替换为的地板 tile source_id（设为 -1 表示直接擦除不替换）
@export var unlock_floor_sid: int = 0

## 门占据的 tile 坐标列表（Layer 1, Vector2i 数组）
## 示例：[Vector2i(56, 23), Vector2i(57, 23), ...]
@export var door_tile_positions: Array[Vector2i] = []

## 门是否在场景加载时就阻挡玩家（有碰撞）
@export var blocks_movement: bool = true

## 解锁后是否播放粒子/震动
@export var play_unlock_fx: bool = true

var _is_unlocked: bool = false
var _tm: TileMap

func _ready() -> void:
    add_to_group("locked_door")
    if tilemap_path:
        _tm = get_node(tilemap_path) as TileMap
    EventBus.door_unlock_requested.connect(_on_unlock_requested)

func _on_unlock_requested(req_door_id: String) -> void:
    if req_door_id != door_id or _is_unlocked:
        return
    _is_unlocked = true
    _play_unlock_sequence()

func _play_unlock_sequence() -> void:
    # 1. 震动（可选）
    if play_unlock_fx:
        CombatFeedback.screen_shake(3.0)
    # 2. 擦除 TileMap 中的锁门 tile，替换为地板
    if _tm:
        for pos in door_tile_positions:
            _tm.erase_cell(1, pos)  # 移除锁门
            if unlock_floor_sid >= 0:
                _tm.set_cell(0, pos, unlock_floor_sid, Vector2i(0, 0))  # 铺地板
    # 3. 禁用自身碰撞
    if blocks_movement and has_node("CollisionShape2D"):
        $CollisionShape2D.set_deferred("disabled", true)
    # 4. 淡出视觉（如果绑定了 sprite）
    if has_node("Sprite"):
        var t := create_tween()
        t.set_process_mode(Tween.TWEEN_PROCESS_IDLE)
        t.tween_property($Sprite, "modulate:a", 0.0, 0.5)
```

**关键点：**
- `door_tile_positions` 是 `Array[Vector2i]`，场景编辑器中可以可视化编辑每个 tile 位置
- 副本设计师在 Godot 编辑器中挂一个 LockedDoor 节点，填上 export 参数，门就准备好了
- `door_id` 与 Objective 的 `completion_action.door_id` 字符串匹配——纯数据对应，零硬编码

### 4.2 LockedDoor 的 TileMap 集成

LockedDoor 本身不负责铺设锁门 tile——那是关卡构建阶段的事（`school_map.gd` 或后续副本的 map 脚本负责）。LockedDoor 只负责"解锁后的 tile 替换"。

**铺锁门 tile 的方式（副本 map 脚本中）：**

```gdscript
// 在任意 map 脚本的 _draw() 或类似方法中：
// 锁门 tile 使用 source 10（暗红褐，有碰撞）
for dx in range(-4, 5):
    _tm.set_cell(1, Vector2i(center_x + dx, door_y), 10, Vector2i(0, 0))
```

不同副本可以使用不同的 source ID 来表示锁门——这由 LockedDoor 的 `lock_source_id` export 参数决定。学校副本用 source 10，工厂副本可以用 source 11（生锈的铁门），太空站可以用 source 12（气密门）——LockedDoor 完全不在乎。

### 4.3 与现有 Door.gd 的关系

现有的 `Door` 类（`scripts/door.gd`）是一个"玩家靠近自动开"的简单门。LockedDoor 是"必须由事件解锁"的门。两者独立，互不依赖。一个场景可以同时有两类门。

| 特性 | Door.gd | LockedDoor |
|------|---------|------------|
| 打开方式 | 玩家靠近（body_entered） | EventBus 信号 |
| 可破坏 | 是（take_damage） | 否（锁门不可通过攻击打开） |
| 配置复杂度 | 低（只需 open_radius + hp） | 中（需 door_id + tile 列表） |
| 适用场景 | 教学楼内门、临时障碍 | 关卡关卡门、守卫门、剧情门 |

## 七、 通用化设计 — 新副本 = 纯 .tres 数据配置

### 7.1 核心承诺：程序写一次，副本配数据

```
┌─────────────────────────────────────────────────────────┐
│  程序层（写一次，不改）                                    │
│  ┌─────────────────────────────────────────────────┐    │

│  │ LockedDoor             — 信号驱动的可配置门        │    │
│  │ ProtectableObject      — HP + 被攻击优先级的物体   │    │
│  │ EventBus               — 全局信号总线             │    │
│  │ MissionChain/Stage/Objective/TriggerConfig — Resource │    │
│  └─────────────────────────────────────────────────┘    │
│                         ▲                               │
│                         │ 纯数据配置                     │

│  ┌─────────────────────────────────────────────────┐    │

│  │ 太空站   ← station_chapter.tres                   │    │
│  │ 神社副本 ← shrine_chapter.tres                    │    │
│  │ ...       ...                                     │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

**唯一需要代码的事情：**
- 每个副本需要一个 **地图脚本**（如 `factory_map.gd`），负责 TileMap 构建和预置敌人
- 每个副本需要一个 **工厂方法**（如 `_create_factory_chapter()`），返回 `MissionChain` 资源

**不需要改的代码：**
- `MissionTriggerManager` — 引擎本身
- `LockedDoor` — 门的行为
- `ProtectableObject` — 保护目标的行为
- `EventBus` — 信号协议
- `MissionChain` / `MissionStage` / `MissionObjective` / `TriggerConfig` — 数据结构

### 7.2 新 Trigger 的通用性矩阵

| Trigger 类型 | 引擎注册 | 适用所有副本 | 参数化维度 |
|-------------|---------|-------------|-----------|
| LOCATION_REACH (0) | 已有 | 是 | zone_id, trigger_mode (enter/exit) |
| KILL_COUNT (1) | 已有 | 是 | enemy_filter (any/elite/boss/ranged), target_value, location_filter |
| TIME_SURVIVE (2) | 已有 | 是 | target_value（纯计时） |
| INTERACT (3) | 已有（未实现） | 是 | interactable_id |
| COLLECT (4) | 已有（未实现） | 是 | item_id, count |
| BOSS_HP_THRESHOLD (5) | 已有（未实现） | 是 | boss_id, hp_percentage |
| **DEFEND_ZONE (7)** | **新增** | 是 | zone_id, pause_on_exit, reset_on_exit |
| **PROTECT_OBJECT (8)** | **新增** | 是 | object_id, target_hp_threshold |

再加上 `completion_action`（不限于特定 Trigger 类型），副本设计师的手牌从 3 张变成 8 种 Trigger x N 种 Action 的组合。

以下不是 GDScript 代码——这是一个纯数据配置的伪代码，展示设计师如何用 `MissionChain` 资源构建新副本。**所有引用的 Trigger 类型、Objective 字段、completion_action 都是已注册的系统原语。**

```
MissionChain "factory_chapter1" {
  chain_name: "锈蚀之地"
```

### 7.4 completion_action 扩展规划

当前 `_dispatch_completion_action()` 只处理 `"unlock_door"` 类型，但分发逻辑用 match 语句实现，扩展只需加 case：

```gdscript
func _dispatch_completion_action(obj: MissionObjective) -> void:
    var action: Dictionary = obj.completion_action
    match action.get("type", ""):
        "unlock_door":
            EventBus.door_unlock_requested.emit(action["door_id"])
        # 未来扩展（本期不实现，仅预留）：
        # "spawn_wave":
        #     EventBus.wave_spawn_requested.emit(action["wave_config_id"])
        # "activate_zone":
        #     EventBus.zone_activated.emit(action["zone_id"])
        # "grant_item":
        #     EventBus.item_granted.emit(action["item_id"])
        # "trigger_dialogue":
        #     EventBus.dialogue_triggered.emit(action["dialogue_id"])
        # "modify_enemy_stats":
        #     EventBus.enemy_stats_modified.emit(action["stat"], action["multiplier"])
```

**所有扩展都不需要改 MissionObjective、MissionStage、MissionChain 的数据结构**——因为 `completion_action` 已经是 Dictionary，新增 type 只是在 match 中加 case + 在 EventBus 中加对应的信号。

### 7.5 LockedDoor 的通用性：多副本、多门实例

一个场景可以有 N 个 `LockedDoor` 实例，各自通过 `door_id` 独立响应：

```
场景树:
  SchoolMap
    ├── LockedDoor (door_id="gym_lock_door")    ← 体育馆入口
    ├── LockedDoor (door_id="gate_lock_door")   ← 校门（如果将来锁校门）
    └── LockedDoor (door_id="office_door")      ← 教学楼某间锁住的办公室

副本链:
  Stage 3 Objective A: KILL_COUNT elite → completion_action{"unlock_door","gym_lock_door"}
  // 未来: Stage X → completion_action{"unlock_door","gate_lock_door"}
```

每个 LockedDoor 实例是独立的——它有自己的 `door_tile_positions`、`lock_source_id`、`unlock_floor_sid`。副本设计师在场景编辑器中手动配置每个门的参数。系统代码零修改。

## 八、 体验审核意见 — Todd Howard

### 玩家自由度评估

**B+ → A- 的提升。** 旧版玩家只有两个动词——"去"和"杀"。新版玩家有了"守"——这是第三个核心动词。更重要的是，DEFEND_ZONE 给了玩家一个"我需要决定"的时刻：*"我要站在走廊中间接敌，还是卡在教室门洞一侧逐个清理？"* 这个决定是玩家自己的，不是我替他做的。

但还可以更好。如果驻守区域的边界不是矩形而是不规则的（比如"走廊+楼梯口"），玩家会有更多地形利用的空间。这个留给关卡设计师。

### 世界沉浸感

**Stage 3（守住走廊）是我最喜欢的设计。** 原因：两边教室是空的——你刚刚在里面杀完了15个人。现在你在走廊里，那些空教室的门洞张开着，黑暗里传来脚步声。*你的背会发凉。*

**门锁也需要沉浸感。** 锁住的门不只是红色的——它应该有一个微弱的红光或暗角，暗示"这里不对"。击败守卫后，门的红光熄灭，恢复成普通门的颜色。这些微小的视觉语言比任何文字都更能让玩家相信这个世界。

### "水冷器时刻"潜力

1. **"我在走廊里守了30秒，最后5秒两边教室同时涌出来，我差点死了——然后门开了。我跑出去的时候还在抖。"**
2. **"那个精英守卫站在体育馆门口不动，我以为能绕过去。不行。必须杀。然后门自己开了。那种锁链碎掉的声音——爽。"**
3. **"我第一次保护那个东西的时候，看见敌人绕过了我去打它。我喊了一声'操！'然后冲回去。"**（PROTECT_OBJECT 的预期效果，本期不做但不妨碍憧憬。）

### 需要打开的限制

1. **DEFEND_ZONE 的 `pause_on_exit` 应该可配置为"重置"。** 有些副本设计可能要求玩家"一次都不出去"——出去就重来。这个选项应该开放给关卡设计师。

2. **驻守期间敌人 AI 可以更聪明。** 不只是向玩家移动，而是从两侧教室绕后（需要门洞之外的路径）。这需要 `NavManager` 扩展——本期不做但记下来。

3. **锁门不应该只有一个精英守卫。** 如果未来副本需要"击败三个精英才能开门"（3 个 KILL_COUNT objective 都带相同的 unlock_door completion_action），系统需要支持"多 Objective 完成 → 门开"。当前设计只支持单个 Objective 完成触发——可以通过 `completion_action` 带 `required_count` 参数扩展。

4. **玩家应该能看到锁与钥匙的关系。** 精英守卫身上应该有一个视觉标记（发光、光环、特殊颜色），让玩家一眼知道"杀了这个东西门会开"。这不需要系统改动，是美术和 enemy_config 的事情。

### 最终评判：🗻 Mountain

这不是一个小改动。这是在给游戏增加第三个维度——从"去→杀"变成"去→杀→守→解锁"。每一个副本设计师拿到这套工具后，可以像搭积木一样组合出完全不同的体验。

但我必须说实话：**`completion_action` 是我们给自己挖的一个宝藏矿洞。** 今天只用了 `unlock_door` 这一个动作。但 Dictionary 结构的扩展性意味着：未来每一个 Objective 完成都能触发一个世界变化——门开了、怪来了、天气变了、BGM 换了。这是"系统涌现"的种子。不要急着实现全部。种下去，让它自己长。

> *"It just works." — 但现在，它还需要你写代码。去写吧。*
