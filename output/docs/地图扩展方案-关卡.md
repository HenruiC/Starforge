## 一、现状分析

### 1.1 当前地图参数（school_map.gd）

```
T  = 48 px（每格）
MW = 3200 px → 67 列
MH = 2400 px → 50 行
总面积 = 7,680,000 px²
```

### 1.2 当前房间尺寸（tile 单位）

| 区域 | 宽度 | 高度 | 面积(tiles) | 面积(px) | 问题 |
|------|------|------|-------------|----------|------|
| 体育馆(Boss间) | 18 | 12 | 216 | 864×576 | **太小** — 4个精英+玩家+Boss挤在500px直径圈内 |
| 左教学楼 | 18 | 18 | 324 | 864×864 | 课桌填充后可用空间约40% |
| 右教学楼 | 18 | 18 | 324 | 864×864 | 同上 |
| 连廊 | 6 | 3 | 18 | 288×144 | 太窄，dodger敌人几乎无法躲避 |
| 操场 | 67 | ~8 | ~536 | 3216×384 | 仅8格深，跑动距离不足 |

### 1.3 核心痛点

1. **Boss间是牢笼** — 18×12 格对俯视角Boss战来说就是个小房间。玩家没有周旋空间，Boss的冲锋/范围技能根本没有设计余地。
2. **教学楼是走廊** — 18×18 减去墙壁和课桌阻塞，实际可移动区域不到8×8。
3. **操场太浅** — Stage 1 跑动距离约384px，玩家出生3秒就能跑到头。
4. **连廊是瓶颈** — 6格宽的通道里塞了8个敌人，纯挨打体验。

## 二、新地图方案

### 2.1 总体参数

```
T  = 48 px（不变，保持现有碰撞/寻路系统兼容）
MW = 5760 px → 120 列（原来 67）
MH = 4320 px →  90 行（原来 50）
总面积 = 24,883,200 px²（原来的 3.24 倍）
```

选择 5760×4320 的理由：
- 120×90 = 10,800 tiles，是原来的 3.24 倍面积
- 每个区域有真正的"空间"，不是"格子"
- 仍然在 Godot TileMap 性能舒适区内（<20,000 tiles）
- 保证 16:12 比例，可以裁剪适配不同屏幕

### 2.2 tile 坐标重新分配

```
常量：
  gx = 60（cols / 2，地图中轴）
  gy = 6（体育馆起始 y）

体育馆 / Boss 间：
  x: gx-12 .. gx+12 → 48..72（24 格宽，原来 18）
  y: 6..23（18 格高，原来 12）
  面积: 24×18 = 432 tiles（原来 216，翻倍）
  实际可用: ~18×14 减去柱子和观众席装饰

连廊（体育馆→教学楼连接通道）：
  x: gx-5 .. gx+5 → 55..65（11 格宽，原来 6）
  y: 24..41（18 格长）
  改用更宽的斜坡式连接，视觉上是从体育馆二楼俯瞰走廊

左教学楼：
  bx = 14, lw = 26（原来 18）
  by = rows - 36 = 54, lh = 22（原来 18）
  面积: 26×22 = 572 tiles，实际可用 ~300 tiles（减去课桌/柱子）

右教学楼：
  rx = gx + 8 = 68, lw = 26（原来 18）
  by = 54, lh = 22
  面积: 572 tiles

连廊（教学楼之间）：
  x: bx+lw .. rx → 40..68（约28格跨距）
  mid_y = by + lh/2 = 54 + 11 = 65
  走廊宽度从3格扩大到5格
  y 范围: 63..67

操场：
  y 范围: by+lh = 76..89（14 格深，原来约8格）
  全宽: 0..119（约120格宽）
  面积: 120×14 = 1680 tiles

校门：
  y = 89（最南端一行）
  中心: gx = 60
  门宽: 11 格（gx-5..gx+5，原来 9 格）
```

### 2.3 新增区域

```
前庭（校门与操场之间）：
  x: gx-15 .. gx+15（31格宽）
  y: 76..88（13格深）
  功能: Stage 1 主战场 + 最终逃离路径
  地面: 浅沙色操场地面，但铺有红色跑道线作为视觉引导

跑道：
  绕操场一圈的红色跑道纹理
  外圈: y=76 全宽 + y=88 全宽
  侧线: x=gx-18 和 x=gx+18，y=77..87
  不仅装饰，还提供天然的 kiting 路径视觉引导

Boss间前厅（体育馆入口过渡区）：
  x: gx-8 .. gx+8（17格宽）
  y: 24..29（6格深）
  功能: Boss 登场序列的发生地，有足够的空间承载登场动画
  地面: 走廊地板（中灰），两侧有倒塌的体育器材（装饰性墙壁tile）
```

### 2.4 房间尺寸对比

| 区域 | 旧尺寸(tiles) | 新尺寸(tiles) | 旧面积(px) | 新面积(px) | 增幅 |
|------|--------------|--------------|-----------|-----------|------|
| Boss间 | 18×12 | 24×18 | 864×576 | 1152×864 | +100% |
| 左教学楼 | 18×18 | 26×22 | 864×864 | 1248×1056 | +89% |
| 右教学楼 | 18×18 | 26×22 | 864×864 | 1248×1056 | +89% |
| 连廊(教室间) | 6×3 | 28×5 | 288×144 | 1344×240 | +678% |
| 连廊(体→教) | 10×8 | 11×18 | 480×384 | 528×864 | +148% |
| 操场 | 67×8 | 120×14 | 3216×384 | 5760×672 | +213% |
| 前庭 | — | 31×13 | — | 1488×624 | 新增 |

## 三、代码变更清单

### 3.1 school_map.gd — 常量

```gdscript
# 修改
const MW := 5760   # 原来 3200
const MH := 4320   # 原来 2400
```

### 3.2 school_map.gd — _draw() 区域坐标

```gdscript
var gx := cols / 2     # 60（原来 33）
var bx := 14            # 左教学楼 x（原来 8）
var by := rows - 36     # 教学楼起始 y = 54（原来 rows-26=24）
var lw := 26            # 教学楼宽度（原来 18）
var lh := 22            # 教学楼高度（原来 18）
var rx := gx + 8        # 右教学楼 x = 68（原来 gx+6）
var mid_y := by + lh / 2  # 65（原来 33）
var gy := 6             # 体育馆 y（不变）
```

### 3.3 school_map.gd — _draw_boundary_walls() 校门位置

```gdscript
# 校门中央11格（原来9格）
for dx in range(-5, 6):
    _tm.set_cell(1, Vector2i(gx + dx, rows - 1), 7, Vector2i(0, 0))
```

### 3.4 school_map.gd — _building() 体育馆调用

```gdscript
# 原来: _building(gx - 9, gy, 18, 12, 6, 0, "", 3, 6)
_building(gx - 12, gy, 24, 18, 6, 0, "", 3, 6)
```

### 3.5 school_map.gd — 体育馆入口

```gdscript

for dx in range(-4, 5):
```

### 3.6 school_map.gd — get_preplaced_enemies() 坐标

所有敌人坐标需要重新设计（见第四节）。当前坐标基于 3200×2400 地图。

### 3.7 school_map.gd — generate_map_texture()


### 3.8 school_map.gd — _draw_floor_zones()

所有区域范围需要按新坐标重写。操场默认地板填充从 `for y in rows` 的 90 行全部覆盖。

## 四、敌人重新布置（68 个预置点位）

### 4.1 操场 + 前庭（Stage 1 — 18个敌人）

分散在 120×14 的广阔区域。采用"松散集群"模式：3-4个一组，组间留30格(约500px)间距。

```
集群A（校门附近，x=40..50, y=80..87）: 4 melee
集群B（操场中左，x=20..35, y=78..85）: 3 melee + 1 ranged
集群C（操场中右，x=75..95, y=78..85）: 3 melee + 1 ranged
集群D（前庭正中，x=55..65, y=78..82）: 2 melee + 1 dodger
集群E（跑道远端，x=10..25, y=76..80）: 2 melee + 1 stationary
```

```
教室前部（x=18..26, y=58..64）: 3 stationary + 2 melee
教室中部（x=20..30, y=64..70）: 3 melee + 1 ranged
教室后部（x=18..26, y=70..75）: 2 melee + 1 ranged
```

## 六、实施风险与检查点

| 风险 | 影响 | 缓解 |
|------|------|------|
| 敌人坐标全部重新手算 | 高 — 68个点位容易出错 | 按集群分批写入，每个集群写完后跑测试 |
| 连廊加宽后导航区域重算 | 中 — NavManager 可能需要对应调整 | NavManager 目前基于 `get_random_nav_point()`，应自动适配 |
| 地图纹理生成耗时 | 低 — 120×90=10800像素，Image.create 毫秒级 | 无 |
| 旧存档/zone坐标不同步 | 中 — mission_manager 的 zone_definitions 是硬编码 | 与地图常量同步更新 |
| 生成地图像素量增大 | 低 — 67×50=3350→120×90=10800像素，仍在合理范围 | Godot Image 操作上万像素无压力 |

## 七、实施顺序

1. **先改常量** — MW/MH + cols/rows 推导式 → 编译通过
2. **再改 _draw()** — 区域坐标全部重算 → 地图可渲染
3. **zone_definitions 同步** — mission_manager.gd 坐标更新
4. **敌人坐标重排** — get_preplaced_enemies() 全部重写
5. **MapViewport 适配** — _build_map_viewport() 的 cols/rows
6. **跑通整局** — 从 CharSelect 到 Boss 击败 → 确认所有触发正常
