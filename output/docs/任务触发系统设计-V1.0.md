# 《天赋猎人》任务触发系统设计文档

版本：V1.0
目标：设计一套数据驱动、可复用的通用任务触发系统，替换当前硬编码的 MissionManager

---

## 一、系统概述

### 1.1 核心概念

将任务系统拆解为三个独立层：

```
┌──────────────────────────────────────┐
│  配置层 (Resource)                    │
│  MissionChain / Stage / Objective    │  ← 策划配置，数据驱动
├──────────────────────────────────────┤
│  运行层 (Manager)                    │
│  MissionTriggerManager               │  ← 运行时评估、状态推进
├──────────────────────────────────────┤
│  表现层 (UI)                         │
│  MissionPromptUI / HUD Objective     │  ← 提示、箭头、完成反馈
└──────────────────────────────────────┘
```

| 术语 | 定义 |
|------|------|
| **MissionChain** | 一个完整副本/关卡的任务链，包含多个 Stage |
| **MissionStage** | 任务链中的一个阶段，包含激活条件 + 若干 Objective |
| **MissionObjective** | 一个具体的任务目标，绑定一个 Trigger 来追踪进度 |
| **Trigger** | 决定 Objective 如何被推进的条件/事件定义 |
| **Evaluator** | 运行时监听游戏事件、判断 Trigger 是否达成的逻辑单元 |

### 1.2 设计目标

1. **数据驱动** — 所有任务配置从 `.tres` Resource 文件加载，策划无需改代码
2. **通用复用** — 同一种 Trigger 类型可用于不同副本、不同阶段、不同 Objective
3. **平滑集成** — 复用现有 EventBus 信号体系，不破坏 GameManager/MissionManager 现有接口
4. **渐进迁移** — 旧 MissionManager 的 `stage_cleared`/`boss_spawned` 信号和 `get_title()`/`get_objectives()` HUD 接口保持不变
5. **扩展友好** — 新增 Trigger 类型只需：添一个枚举值 + 注册一个 Evaluator 函数

### 1.3 在整体游戏中的定位

```
                    ┌──────────────┐
                    │  EventBus    │
                    │  (全局事件)   │
                    └──────┬───────┘
        ┌──────────────────┼──────────────────┐
        │                  │                  │
   enemy_killed      player_moved?      wave_changed
        │                  │                  │
        ▼                  ▼                  ▼
┌─────────────────────────────────────────────────┐
│          MissionTriggerManager                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │Location   │  │Kill      │  │Time      │     │
│  │Evaluator  │  │Evaluator │  │Evaluator │ ... │
│  └──────────┘  └──────────┘  └──────────┘     │
│         │             │             │          │
│         └─────────────┼─────────────┘          │
│                       ▼                        │
│              ┌────────────────┐                │
│              │ Stage 推进逻辑  │                │
│              └───────┬────────┘                │
└──────────────────────┼──────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
   ┌──────────┐ ┌──────────┐ ┌──────────────┐
   │HUD 更新   │ │提示UI    │ │对话/剧情触发  │
   │(现有接口) │ │(新增)    │ │(现有Dialogue) │
   └──────────┘ └──────────┘ └──────────────┘
```

## 二、核心规则

### 2.1 Trigger 类型枚举

```
enum MissionTriggerType {
    LOCATION_REACH    = 0   // 玩家到达指定区域
    KILL_COUNT        = 1   // 击杀 N 个敌人
    TIME_SURVIVE      = 2   // 存活 X 秒
    INTERACT          = 3   // 玩家按键交互（预留）
    COLLECT           = 4   // 收集 N 个物品（预留）
    BOSS_HP_THRESHOLD = 5   // Boss 血量低于阈值（预留）
}
```

**当前 V1.0 实现前三类，后三类定义接口但不实现逻辑。**

### 2.2 每种 Trigger 的参数定义

#### 2.2.1 LOCATION_REACH — 位置触发

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `zone_id` | String | 是 | 目标区域标识符，与场景中 Area2D 的 `zone_id` 元数据对应 |
| `trigger_mode` | String | 否 | `"enter"`（进入即触发，默认）/ `"exit"`（离开触发）/ `"stay"`（持续在内，按秒累计） |
| `stay_duration` | float | 否 | 仅 `trigger_mode="stay"` 时有效，需在区域内停留的秒数 |

**场景集成规范**：场景中用于位置触发的 Area2D 节点必须：
1. 加入 `"mission_zone"` 组
2. 在 `meta` 中设置 `zone_id` 为对应的字符串标识
3. 使用 `body_entered` / `body_exited` 信号连接至 MissionTriggerManager

#### 2.2.2 KILL_COUNT — 击杀触发

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `target_count` | int | 是 | 需要击杀的数量 |
| `enemy_filter` | String | 否 | 敌人筛选：`"any"`（任意，默认）/ `"elite"`（精英）/ `"boss"`（Boss）/ `"ranged"`（远程） |
| `location_filter` | String | 否 | 区域筛选：仅击杀指定 `zone_id` 内的敌人才计数，空字符串 = 不限区域 |

**事件源**：监听 `EventBus.enemy_killed` 信号，结合 Enemy 节点的 `is_elite`/`is_boss`/`is_ranged` 属性判断。

**扩展示例**："在教室区域击杀 5 个精英怪" → `enemy_filter="elite"`, `location_filter="classroom_zone"`

#### 2.2.3 TIME_SURVIVE — 时间触发

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `survive_seconds` | float | 是 | 需要存活的秒数 |
| `countdown_from` | String | 否 | 计时起点：`"stage_start"`（本阶段开始时，默认）/ `"game_start"`（游戏开始）/ `"trigger"`（由其他事件启动） |
| `pause_on_dialogue` | bool | 否 | 对话/暂停期间是否暂停计时，默认 `true` |

**注意**：`TIME_SURVIVE` 的 "survive" 意味着玩家死亡则失败。若需 "坚持 X 秒后胜利"，应配合 `is_stage_fail_on_death` 标志。

### 2.3 Stage 推进规则

```
激活条件满足 ──→ Stage 进入 Active 状态
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
      Obj 1       Obj 2       Obj 3    (并行追踪)
          │           │           │
          └───────────┼───────────┘
                      ▼
               全部必填 Obj 完成
                      │
                      ▼
              Stage 进入 Completed 状态
                      │
              ┌───────┴───────┐
              ▼               ▼
       下一 Stage 激活      Chain 结束
```

**规则**：
1. 同一 Stage 内的 Objectives **并行**追踪（所有 Objective 同时激活）
2. 必填 Objective（`is_optional=false`）全部完成后 Stage 才算完成
3. 可选 Objective（`is_optional=true`）完成时给额外奖励但不阻塞推进
4. Stage 完成后立即激活下一个 Stage（`next_stage_id` 指向），或结束 Chain

### 2.4 与 GameState 的耦合规则

| GameState | 任务系统行为 |
|-----------|-------------|
| `PLAYING` | 正常追踪、计时 |
| `PAUSED` / `MAP` / `INVENTORY` | 暂停所有计时类 Trigger（`pause_on_dialogue=true` 时） |
| `DIALOGUE` | 暂停计时，但保留进度（对话可能被任务触发） |
| `DEAD` | 停止所有追踪，记录失败状态 |
| `CHAR_SELECT` | 未初始化，不追踪 |

## 三、详细设计

### 3.1 数据结构

#### 3.1.1 MissionChain（任务链 — Resource）

```
class_name MissionChain
extends Resource

字段：
  chain_id: String              # 唯一标识，如 "school_chapter1"
  chain_name: String            # 显示名称，如 "废校求生"
  stages: Array[MissionStage]   # 阶段列表，按数组顺序执行
  is_repeatable: bool           # 是否可重复挑战，默认 false
  on_chain_complete: String     # 完成后的行为: "none" / "show_score" / "unlock_next"
```

#### 3.1.2 MissionStage（任务阶段 — Resource）

```
class_name MissionStage
extends Resource

字段：
  stage_id: int                 # 阶段编号（1-based）
  stage_title: String           # 阶段标题，如 "第一试炼 · 踏入废校"
  narrative: NarrativeConfig    # 叙事配置（对话触发等）
  activation_condition: TriggerConfig   # 激活本阶段的条件（首阶段为 null）
  objectives: Array[MissionObjective]   # 本阶段目标列表
  on_activate: PromptConfig     # 激活时的 UI 提示
  on_complete: PromptConfig     # 完成时的 UI 提示
  dialogue_trigger: DialogueTriggerConfig  # 阶段切换时的对话
  spawn_override: SpawnOverrideConfig     # 可选：本阶段敌人出生配置覆写
```

#### 3.1.3 MissionObjective（任务目标 — Resource）

```
class_name MissionObjective
extends Resource

字段：
  objective_id: String          # 唯一标识，如 "kill_20_enemies"
  description: String           # 显示文本，如 "击杀20个怪物"
  hint: String                  # 小提示，如 "走廊窄，注意走位"
  trigger: TriggerConfig        # 追踪条件
  is_optional: bool             # 是否为可选目标，默认 false
  show_progress_bar: bool       # 进度是否用数值展示（true=显示 "15/20"，false=仅显示 ✓/□）
  reward_on_complete: Dictionary # 完成奖励 {"xp": 50, "heal": 20} 等
```

#### 3.1.4 TriggerConfig（触发条件 — Resource）

```
class_name TriggerConfig
extends Resource

字段：
  trigger_type: MissionTriggerType
  params: Dictionary            # 类型相关的参数，见 2.2 节各类型定义
  # 通用参数
  target_value: float           # 目标值（count / seconds / threshold）
  current_value: float          # 当前值（运行时更新，初始为 0）
```

`params` 的键值规范（按类型）：

| trigger_type | params 键 |
|-------------|-----------|
| LOCATION_REACH | `zone_id`, `trigger_mode`, `stay_duration` |
| KILL_COUNT | `enemy_filter`, `location_filter` |
| TIME_SURVIVE | `countdown_from`, `pause_on_dialogue` |
| INTERACT | `interact_target_id`, `prompt_text` |
| COLLECT | `item_id` |
| BOSS_HP_THRESHOLD | `boss_id`, `threshold_pct` |

#### 3.1.5 PromptConfig（提示配置 — Resource）

```
class_name PromptConfig
extends Resource

字段：
  text: String                  # 提示文字，如 "前往教学楼"
  text_color: Color             # 文字颜色，默认 Color.WHITE
  show_arrow: bool              # 是否显示方向箭头
  arrow_target: Vector2         # 箭头指向的世界坐标（show_arrow=true 时必填）
  arrow_target_zone: String     # 备选：指向 zone_id 的中心位置（与 arrow_target 二选一）
  display_duration: float       # 显示时长（秒），0 = 直到 Stage 完成
  animation: String             # 动效类型: "slide_down" / "fade_in" / "pulse" / "typewriter"
  priority: int                 # 优先级（多个提示同时触发时，数字越大越优先显示），默认 0
  sound_id: String              # 可选：音效 ID，空字符串 = 无音效
```

#### 3.1.6 完整数据关系图

```
MissionChain (1)
  └── MissionStage (1..N)
        ├── activation_condition: TriggerConfig
        ├── objectives (1..N): MissionObjective
        │     └── trigger: TriggerConfig
        ├── on_activate: PromptConfig
        └── on_complete: PromptConfig
```

### 3.2 MissionTriggerManager 设计

#### 3.2.1 职责

- 加载和持有当前 `MissionChain` Resource
- 管理当前 Stage 和 Objective 的运行时状态
- 注册/注销各类型 Evaluator
- 向 HUD 和 Prompt UI 推送更新
- 发出 `stage_activated` / `stage_completed` / `chain_completed` 信号

#### 3.2.2 信号定义

```
signal stage_activated(stage_id: int)
signal stage_completed(stage_id: int)
signal objective_progressed(stage_id: int, objective_id: String, current: float, target: float)
signal objective_completed(stage_id: int, objective_id: String)
signal chain_completed(chain_id: String)
signal prompt_requested(prompt: PromptConfig)      # 请求显示提示
signal prompt_dismissed(prompt_id: String)          # 请求隐藏提示
```

#### 3.2.3 Evaluator 注册机制

每个 Trigger 类型对应一个 Evaluator 函数，在 `MissionTriggerManager` 初始化时注册：

```
var _evaluators: Dictionary = {
    MissionTriggerType.LOCATION_REACH: _eval_location,
    MissionTriggerType.KILL_COUNT: _eval_kill,
    MissionTriggerType.TIME_SURVIVE: _eval_time,
}
```

新增 Trigger 类型时：
1. 在枚举中添加值
2. 实现 `_eval_xxx(trigger: TriggerConfig, delta: float) -> void` 函数
3. 在 `_evaluators` 字典中注册

#### 3.2.4 运行时状态机

```
Stage 状态流转:
  PENDING ──activation_condition满足──→ ACTIVE ──所有必填Obj完成──→ COMPLETED
                                              │
                                              └── 玩家死亡 ──→ FAILED

Objective 状态流转:
  PENDING ──Stage进入ACTIVE──→ TRACKING ──trigger.current>=trigger.target──→ COMPLETED
```

#### 3.2.5 与现有代码的集成点

| 现有组件 | 集成方式 | 变更程度 |
|---------|---------|---------|
| `EventBus` | 新增信号：`player_entered_zone(zone_id)`、`player_exited_zone(zone_id)`、`enemy_killed_filtered(pos,score,is_elite,is_boss,is_ranged,kill_zone)` | 小 — Enemy 的 `_die()` 方法扩展传参 |
| `GameManager` | `_mission_manager` 改为持有 `MissionTriggerManager` 实例；`_update_mission_hud()` 调用新接口；`_on_stage_cleared()` 保留 | 中 — 替换 `_MissionManagerScript` 引用 |
| `GameState` | `state_changed` 信号连接至 `MissionTriggerManager`，用于暂停/恢复计时 | 小 — 新增一行信号连接 |
| `MissionManager`（旧） | 保留为兼容层或直接替换；新系统实现相同的 `get_title()` / `get_objectives()` / `get_current_stage()` 接口 | — |
| `DialoguePanel` | 通过 `stage_activated` / `stage_completed` 信号触发对话，替代当前 `_trigger_stage_dialogue()` 中的硬编码 `match stage` | 中 — 对话数据从 MissionStage 读取 |

#### 3.2.6 迁移路径（从旧 MissionManager → 新系统）

```
阶段1（共存）: 新 MissionTriggerManager 与旧 MissionManager 并行运行
              GameManager 优先使用新系统，旧系统保留但不再调用
阶段2（切换）: GameManager 完全切换到新系统
              旧 MissionManager 文件归档为 mission_manager_legacy.gd
阶段3（清理）: 删除旧 MissionManager
              确认所有功能正常
```

### 3.3 提示 UI 交互规范

#### 3.3.1 MissionPromptUI 组件

一个新的轻量级节点，挂载在 HUDLayer 下，不依赖场景中的复杂 UI 结构。

```
MissionPromptUI (Control)
  ├── PromptContainer (VBoxContainer)
  │     ├── StageActivatePrompt (Label)    # 阶段激活提示
  │     └── ObjectiveUpdatePrompt (Label)  # 目标更新提示
  └── DirectionArrow (Label)               # 方向箭头，独立于容器
```

**设计原则**：所有提示都是动态创建的 Label + Tween 动画，不使用 `.tscn` 场景，与现有 UIEffects 体系一致。

#### 3.3.2 出现时机

| 事件 | UI 行为 | 优先级 |
|------|---------|--------|
| Stage 激活 | 显示 `on_activate` 提示（含可选箭头），持续 `display_duration` 秒后自动消失 | 1（最高） |
| Objective 进度更新 | 右上角弹出 Objective 进度条/数字，1.5 秒后淡出 | 2 |
| Objective 完成 | 该 Objective 文字变绿，打勾动画，0.5 秒后缩回 | 2 |
| Stage 完成 | 屏幕中央显示 `on_complete` 提示（"✓ 阶段完成"），持续 2 秒 | 1 |
| Chain 完成 | 屏幕中央大标题 + 统计信息 | 0（覆盖所有） |
| 多个提示同时触发 | 按优先级排队，高优先级打断低优先级 | — |

#### 3.3.3 位置规范

```
┌──────────────────────────────────────┐
│          [Stage 激活提示]             │  ← 屏幕顶部居中，y = 80
│          "前往教学楼"                 │
│               ↓                      │  ← 方向箭头（在提示下方）
│                                      │
│                                      │
│                    ┌──────────────┐  │
│                    │ Obj 进度更新  │  │  ← 屏幕右侧，y = 200，向下堆叠
│                    │ ✓ 击杀 15/15 │  │
│                    └──────────────┘  │
│                                      │
│                                      │
│              ┌──────────────┐        │
│              │ ✓ 阶段完成    │        │  ← 屏幕中央偏上，y = 300
│              └──────────────┘        │
│                                      │
│  [HUD 任务面板]                      │  ← 屏幕左上角（保持现有位置）
│  第一试炼 · 踏入废校                 │
│  ✓ 抵达中央庭院                      │
│  □ 击退第一波怪物 (5/8)              │
└──────────────────────────────────────┘
```

#### 3.3.4 动效规范

参考现有 `UIEffects` 的参数体系：

| 动效 | 参数 | 说明 |
|------|------|------|
| **Slide Down** | `y: -20 → 0, alpha: 0 → 1, dur=0.3s, ease=OUT_BACK` | Stage 激活提示入场 |
| **Fade Out** | `alpha: 1 → 0, dur=0.25s` | 提示退场 |
| **Pop In** | `scale: 0.8 → 1.15 → 1.0, dur=0.2s, trans=BACK` | Objective 进度弹入 |
| **Checkmark** | `scale: 0 → 1.2 → 1.0, color: WHITE → GREEN, dur=0.3s` | Objective 完成标记 |
| **Stage Complete** | `scale: 0.5 → 1.3 → 1.0, alpha: 0 → 1, dur=0.4s, trans=BACK` | 阶段完成大字 |
| **Pulse** | `scale: 1.0 → 1.05 → 1.0, loop=3, dur=0.5s` | 持续提示呼吸效果 |
| **Typewriter** | `visible_characters: 0 → len, speed=0.02` | 对话风格提示（复用 TYPEWRITE_SPEED） |

全部动画使用 `TWEEN_PROCESS_IDLE` 以保证暂停时仍可播放。

#### 3.3.5 方向箭头的计算规则

```
1. 获取箭头目标世界坐标 (target_pos)
2. 计算从玩家到目标的方向向量: dir = (target_pos - player_pos).normalized()
3. 判断目标是否在屏幕内:
   - 如果在屏幕内 → 箭头指向屏幕内目标位置（跟随目标）
   - 如果不在屏幕内 → 箭头吸附到屏幕边缘:
     - 计算方向与屏幕矩形的交点
     - 将箭头定位在边缘交点上
4. 箭头旋转角度 = atan2(dir.y, dir.x)
5. 箭头 Label 文本 = "▼" (通过旋转来指方向)
6. 每帧更新位置和角度（在 _process 中）
```

性能注意：如果同一时间只有一个箭头，每帧计算开销可忽略。如果担心性能，限制为每 0.1 秒更新一次。

## 四、配置示例：学校副本任务链

### 4.1 任务链概览

```
Chain: school_chapter1 ("废校求生")

Stage 1: 踏入废校
  ├── 激活：进入校门区域
  ├── 目标：抵达中央庭院 (LOCATION_REACH: "zone_courtyard")
  ├── 目标：击退第一波怪物 (KILL_COUNT: target=8)
  └── 提示："前往教学楼" + 方向箭头指向教学楼入口

Stage 2: 清理教室
  ├── 激活：Stage 1 完成
  ├── 目标：击杀 20 个怪物 (KILL_COUNT: target=20)
  ├── 目标：清理 8 张课桌 (COLLECT: target=8) [可选]
  ├── 目标：探索左右两栋教学楼 (LOCATION_REACH: zone_id="both_wings")
  └── 提示："清理教室区域"

Stage 3: 击败体育老师
  ├── 激活：Stage 2 完成 + 到达 Boss 间
  ├── 目标：击败精英怪 (KILL_COUNT: target=1, enemy_filter="elite")
  └── 提示："击败体育老师" + 箭头指向连廊
```

### 4.2 Resource 配置伪数据（.tres 等效表示）

```
# school_chapter1.tres
```

## 五、扩展性设计

### 5.1 新副本复用流程

策划配置一个新副本（如 "废弃医院"）只需以下步骤：

```
1. 创建 hospital_chapter1.tres  →  配置 MissionChain
2. 在场景中放置 Area2D Zone 节点  →  设置 zone_id 元数据
3. 在 GameManager 中加载对应 Chain  →  一行代码: mission_manager.load_chain("hospital_chapter1")
```

**无需修改任何 .gd 脚本代码。**

### 5.2 新增 Trigger 类型模板

假设需要添加 `BOSS_HP_THRESHOLD` 触发：

```
步骤 1：枚举添加
  MissionTriggerType 中新增 BOSS_HP_THRESHOLD = 5

步骤 2：定义 params 规范
  params = {
      "boss_id": String,        # Boss 的标识符
      "threshold_pct": float,   # 血量百分比阈值（如 0.5 = 50%）
      "trigger_once": bool,     # 是否只触发一次，默认 true
  }

步骤 3：实现 Evaluator 函数
  func _eval_boss_hp(trigger: TriggerConfig, _delta: float) -> void:
      var boss = _find_boss_by_id(trigger.params["boss_id"])
      if boss == null: return
      var hp_pct = float(boss._health) / float(boss.max_health)
      if hp_pct <= trigger.params["threshold_pct"]:
          trigger.current_value = 1.0  # 达到阈值
      # 如果需要多次触发（如75%/50%/25%），使用多个 Objective

步骤 4：注册
  _evaluators[MissionTriggerType.BOSS_HP_THRESHOLD] = _eval_boss_hp

步骤 5：监听事件
  EventBus.boss_hp_changed.connect(_on_boss_hp_changed)  # 需要在 EventBus 中新增此信号
```

### 5.3 跨副本共享的 Trigger

某些 Trigger 配置可能在多个副本中复用（如 "击杀 10 个任意敌人"），建议：

1. 创建 `common_triggers/` 目录存放通用的 TriggerConfig Resource
2. 副本的 MissionObjective 直接引用这些 Resource
3. 修改通用 Resource 将影响所有引用它的副本（需谨慎）

### 5.4 调试与测试支持

```
MissionTriggerManager 提供以下调试方法（开发期使用，正式版关闭）：

  debug_print_chain()              # 打印当前任务链完整状态
  debug_force_complete_stage(id)   # 强制完成指定阶段
  debug_force_progress(obj_id, v)  # 强制推进指定目标进度
  debug_list_active_triggers()     # 列出所有活跃的 Trigger 及当前值
  debug_set_time_scale(scale)      # 加速/减速计时器（测试时间触发）
```

建议通过 Godot 的 `--debug` 命令行参数或编辑器调试面板控制开关。

### 5.5 与后续系统的扩展接口

| 未来系统 | 集成方式 | 需要预留 |
|---------|---------|---------|
| 存档系统 | MissionTriggerManager 提供 `serialize()` / `deserialize()` 方法 | 所有运行时状态可序列化 |
| 成就系统 | 监听 `objective_completed` / `chain_completed` 信号 | 无需额外预留 |
| 多人模式 | 每个玩家的 Trigger 独立追踪，共享 Stage 状态 | Trigger 需关联 `player_id` |
| 动态难度 | SpawnOverrideConfig 根据玩家表现动态调整 | 已在 MissionStage 中预留 |
| 剧情分支 | `on_complete.next_stage_id` 支持条件跳转 | 需扩展为条件表达式 |

## 六、数据结构总结

### 6.1 核心数据字段

```gdscript
# === TriggerConfig ===
@export var trigger_type: int = 0           # MissionTriggerType 枚举值
@export var params: Dictionary = {}         # 类型相关参数
@export var target_value: float = 1.0       # 目标值

# === MissionObjective ===
@export var objective_id: String = ""
@export var description: String = ""
@export var hint: String = ""
@export var trigger: TriggerConfig
@export var is_optional: bool = false
@export var show_progress_bar: bool = true
@export var reward_xp: int = 0

# === MissionStage ===
@export var stage_id: int = 0
@export var stage_title: String = ""
@export var activation_condition: TriggerConfig  # null = 自动激活
@export var objectives: Array[MissionObjective] = []
@export var on_activate_prompt: PromptConfig
@export var on_complete_prompt: PromptConfig
@export var dialogue_messages: Array[Dictionary] = []  # {speaker, text}

# === MissionChain ===
@export var chain_id: String = ""
@export var chain_name: String = ""
@export var stages: Array[MissionStage] = []
@export var is_repeatable: bool = false

# === PromptConfig ===
@export var text: String = ""
@export var text_color: Color = Color.WHITE
@export var show_arrow: bool = false
@export var arrow_target: Vector2 = Vector2.ZERO
@export var arrow_target_zone: String = ""
@export var display_duration: float = 0.0
@export var animation: String = "slide_down"
@export var priority: int = 0
@export var sound_id: String = ""
```

### 6.2 状态流转

```
MissionChain:  IDLE → RUNNING → COMPLETED (或 FAILED)

MissionStage:
  IDLE → PENDING → ACTIVE → COMPLETED
                      ↘ FAILED (玩家死亡)

MissionObjective:
  IDLE → PENDING → TRACKING → COMPLETED
                              ↘ SKIPPED (可选目标，阶段其他目标已完成)

TriggerConfig (运行时):
  current_value: 0.0 ──Evaluator逐步推进──→ target_value
  达到 target_value 时 → 所属 Objective 标记 COMPLETED
```

## 七、边界情况与 FAQ

### 7.1 边界情况

| 情况 | 处理方式 |
|------|---------|
| 玩家在 Stage 激活前就到达了目标区域 | `LOCATION_REACH` Evaluator 在 Objective 进入 TRACKING 状态后才开始检测。激活前的位置信息不追溯 |
| 同一个击杀满足多个 KILL_COUNT 条件 | 每个活跃的 `KILL_COUNT` Evaluator 独立计数，一次击杀可同时推进多个 Objective |
| 计时器在暂停/对话期间 | 通过 `GameState.state_changed` 信号控制。`pause_on_dialogue=false` 的计时器继续走 |
| Stage 完成后仍然有活跃的 Objective | 可选 Objective 继续追踪（可后续补做），必填 Objective 强制标记完成 |
| 任务链配置错误（如循环引用） | 加载时做静态校验：检测 stage_id 循环依赖、trigger type 与 params 不匹配等，打印错误并阻止运行 |
| 玩家在任务链中途死亡 | 发送 `chain_completed("failed")`，所有 Stage 标记为 FAILED。重开后从头开始 |
| Zone 的 Area2D 被销毁或不可达 | 位置 Evaluator 超时检查（如超过 300 秒未触发且玩家未接近该区域）→ 在 HUD 显示 "?" 警告 |
| 同一帧内多个 Objective 同时完成 | 按 Objective 数组顺序依次处理，提示 UI 排队显示（间隔 0.3 秒） |
| Display duration = 0 的提示何时消失 | 持续到所属 Stage 完成后 1 秒自动消失 |

### 7.2 FAQ

**Q: 为什么不直接用 Godot 的 Signal 做 Trigger？**
A: Signal 是事件通知机制，Trigger 需要的是条件评估机制。用 Signal 直接做 Trigger 会导致：
- 条件组合困难（"击杀 5 个精英 且 在教室内"）
- 无法做暂停/恢复
- 不可序列化。封装为 TriggerConfig + Evaluator 解决了这些问题。

**Q: 为什么不用 Area2D 的 `body_entered` 直接连 Objective？**
A: Area2D 只知道"有东西进入了"，不知道当前哪个 Stage 需要它、是否需要累计停留时间。在 Evaluator 层做中转可以：
- 按 Stage 状态决定是否响应
- 支持 `stay` 模式（需累计停留 N 秒）
- 支持条件过滤（"只在 Stage 3 激活时有效"）

**Q: 如果两个 Zone 重叠，玩家同时进入怎么办？**
A: 所有匹配的 `LOCATION_REACH` Evaluator 各自独立判断，同时推进各自的 Objective。两个 Zone 重叠是关卡设计问题，系统不做合并处理。

**Q: 旧 MissionManager 的 `notify_kill` / `notify_desk` 等方法如何处理？**
A: 这些手动通知方法全部废弃。新的 Evaluator 直接监听 EventBus 信号，无需 GameManager 手动桥接。迁移期间可以保留方法体但标记为 `@deprecated`。

**Q: 提示箭头的性能开销？**
A: 箭头每帧计算一次 `atan2` + 屏幕边缘裁剪（约 20 次浮点运算），在 60fps 下开销 <0.01ms。如果需要优化，降为每 3 帧更新一次即可。

## 八、设计检查清单

- [x] 核心循环闭环：激活 → 追踪 → 完成 → 下一阶段
- [x] 玩家每个行为有明确反馈：激活提示 / 进度更新 / 完成标记 / 箭头指引
- [x] 系统间无耦合冲突：TriggerManager 通过 EventBus 解耦，不直接依赖 Enemy/Player
- [x] 清晰的成长路径：Stage 串行推进，Objective 并行可选
- [x] 数值空间有余地：`target_value` 使用 float，支持分数/百分比进度
- [x] 无死循环或不可达状态：静态校验 + 超时检查 + 可选目标可跳过
- [x] 配置与逻辑分离：Resource 文件存储配置，Evaluator 函数执行逻辑
- [x] 渐进迁移路径：新旧系统共存 → 切换 → 清理
