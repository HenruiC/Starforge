## 二、小地图实现方案

### 方案 A：SubViewport + Camera2D（官方标准方案）

这是 Godot 社区公认的 2D 小地图最佳实践，也是 Godot Asset Library 中 Mini Map 插件的实现方式。

#### 节点结构

```
CanvasLayer (UI 顶层)
└── MiniMapPanel (Panel / ColorRect — 边框/背景)
    └── SubViewportContainer (SVC, stretch=true)
        └── SubViewport (SV, size=256x256)
            ├── MinimapCamera (Camera2D, zoom=0.1)
            └── PlayerMarker (Sprite2D — 可选)
```

#### 核心机制：共享 World2D

SubViewport 默认拥有独立的 World2D，所以看不到主世界的 TileMap 和玩家。关键一行代码：

```gdscript
# 在 SubViewport 的 _ready() 中
world_2d = get_tree().root.get_viewport().world_2d
```

这样 MinimapCamera 就看进了主世界的同一个 World2D，无需复制任何节点。

#### MinimapCamera 跟踪玩家

```gdscript
extends Camera2D

@export var target: Node2D

func _process(_delta: float) -> void:
    if target:
        global_position = target.global_position
```

#### 关键参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| Camera2D.zoom | (0.08, 0.08) | 5760x4320 地图装进 256x256 SubViewport 约需 zoom=0.044，建议 0.08 显示更清晰的局部 |
| SubViewport.size | (256, 256) | 小地图分辨率 |
| SubViewportContainer.stretch | true | 填满父容器 |
| SubViewport.render_target_update_mode | Always | 实时更新（或用 When Visible 省性能） |

#### Render Layer 过滤

如果不想在小地图上显示敌人/粒子效果：

1. `MinimapCamera.cull_mask` 只勾选特定 layer（如 layer 1, 2）
2. 主世界节点的 `visibility_layer` 按需设置
3. PlayerMarker（Sprite2D 放在 SubViewport 内）的 visibility_layer 纳入 cull_mask

#### 像素美术适配

```gdscript
# SubViewport 上设置
snap_2d_transforms_to_pixel = true
snap_2d_vertices_to_pixel = true
# 在 SubViewportContainer 或 TextureRect 上设置
texture_filter = CanvasItem.TEXTURE_FILTER_NEAREST
```

#### 对本项目的适用性

优点：
- 不引入外部插件，纯 Godot 内置节点
- 与现有 TileMap / ColorRect 体系完全兼容
- PlayerMarker 可以用小的 ColorRect 代替贴图

需要调整：
- 现有 `MapSystem` 用 Label 做 PlayerMarker，需要迁移到 SubViewport 内的 Sprite2D
- 当前地图面板（MapPanel）在 CanvasLayer 中，需要重构节点结构
- 大缩放比下 ColorRect 地板可能看不清——建议小地图只显示区域级大色块


### 方案 B：静态纹理 + 手动绘制（现有方案升级）

当前项目已有一个简化版：`MapSystem` 使用预生成的 `ImageTexture`（由 `school_map.generate_map_texture()` 产生）显示在 `TextureRect` 中，并用 Label 标记玩家位置。

#### 可改进点

1. 将 Label-based 玩家标记改为在 TextureRect 上方覆盖 Sprite2D 或 CanvasItem
2. 地图纹理增大分辨率（当前 120x90 像素，可提升到 480x360 或更高）
3. 支持滚动/缩放地图面板

#### 比对 SubViewport 方案

| 维度 | SubViewport 方案 | 静态纹理方案 |
|------|-----------------|-------------|
| 实时性 | 可见地图上所有动态物体 | 只能看到预生成的静态地形 |
| 性能 | 额外渲染一遍 | 仅一张静态纹理 |
| 开发量 | 需要重构面板 | 在现有基础上增量 |
| 视觉效果 | 更好（缩放平滑、任意精度） | 受限于纹理分辨率 |

## 三、战争迷雾实现方案

### 方案 C：ImageTexture 逐个像素更新（社区主流方案）

这是 Merxon22 教程和社区讨论中最推荐的方案，也是本项目 `MapSystem` 已经在用的方向。

#### 核心原理

1. 创建一个 `Image`，尺寸对应地图的 Tile 网格（或更细粒度的像素网格）
2. 用 `Image.set_pixel()` / `Image.fill_rect()` 在 GDScript 中更新像素
3. 用 `ImageTexture.update(image)` 推送到 GPU（复用现有纹理内存，高效）
4. 将 `ImageTexture` 赋值给覆盖在 TileMap 上方的 `TextureRect` 或 `Sprite2D`

#### 基本代码

```gdscript
# 1. 创建迷雾 Image（与地图 tile 网格对应）
var fog_grid_width := 120   # 5760 / 48
var fog_grid_height := 90   # 4320 / 48
var fog_image := Image.create(fog_grid_width, fog_grid_height, false, Image.FORMAT_RGBA8)

# 初始全黑（未探索）
fog_image.fill(Color(0, 0, 0, 0.7))  # 0.7 不透明度

# 2. 创建 ImageTexture 并赋给覆盖层
var fog_texture := ImageTexture.create_from_image(fog_image)
$FogOverlay.texture = fog_texture
# 关键：用 NEAREST 过滤避免像素间混合
$FogOverlay.texture_filter = CanvasItem.TEXTURE_FILTER_NEAREST

# 3. 每帧/定时更新可见区域
func reveal_area(world_pos: Vector2, radius: int) -> void:
    var cx := int(world_pos.x / TILE_SIZE)
    var cy := int(world_pos.y / TILE_SIZE)
    for dx in range(-radius, radius + 1):
        for dy in range(-radius, radius + 1):
            if dx * dx + dy * dy > radius * radius:
                continue
            var nx := cx + dx
            var ny := cy + dy
            if nx >= 0 and nx < fog_grid_width and ny >= 0 and ny < fog_grid_height:
                fog_image.set_pixel(nx, ny, Color(0, 0, 0, 0.0))  # 完全透明 = 已探索
    fog_texture.update(fog_image)  # 关键：用 update() 而非 recreate
```

#### 性能考虑

| 要素 | 评估 |
|------|------|
| `Image.set_pixel()` 每帧调用 | 轻量，~300 像素/帧无压力 |
| `ImageTexture.update()` | GPU 端纹理上传，120x90 RGBA8 ≈ 43KB，极快 |
| `FORMAT_RGBA8` | 适合半透明迷雾（已探索但不在视线内用灰色，未探索用黑色） |
| 最大尺寸限制 | 16384x16384（硬件限制），远大于本项目的 120x90 |

#### 多层迷雾支持

可以用多个 Image / TextureRect 实现（如文明系列的：
- **未探索（黑色不透）** — `Color(0, 0, 0, 1.0)`
- **已探索但不在视野（灰色半透）** — `Color(0, 0, 0, 0.45)`
- **当前视野（完全透明）** — `Color(0, 0, 0, 0.0)`

用一个 Image 的不同 alpha 值即可表达三种状态。

#### 对本项目的适用性

优点：
- 本项目 `MapSystem._draw_fog()` 已在使用 Image + ImageTexture，方向正确
- 与 ColorRect 纯色体系无缝集成
- 不依赖 shader（团队可能 shader 经验不足时的安全选择）

当前问题（需修复）：
- `_draw_fog()` 每帧 `Image.create()` + `ImageTexture.create_from_image()`，严重浪费。应改为：初始化一次 Image/ImageTexture，后续只用 `set_pixel()` + `update()`
- `_fog_grid` 用 `Array[Array]` 存 `bool`，可以简化为直接用 `Image.get_pixel()` 读取颜色判断


### 方案 D：Visibility Mask Shader（GPU 加速方案）

从 godotshaders.com 的 Hexagon TileMap Fog of War 移植来的方案，适合需要更平滑视觉效果或更大规模地图的场景。

#### 核心原理

1. 将迷雾 ImageTexture 传给 TileMap 上挂载的 `ShaderMaterial`
2. Shader 在 GPU 端采样 visibility mask，决定每个 tile/fragment 的颜色
3. CPU 端仅需更新一个 Image 的像素，GPU 自动完成所有 tile 的着色

#### Shader 示例（方形 TileMap 适配版）

```glsl
shader_type canvas_item;
render_mode unshaded;

uniform sampler2D visibility_mask : filter_nearest, repeat_disable;
uniform vec2 mask_size = vec2(120.0, 90.0);
uniform vec2 map_offset = vec2(0.0, 0.0);

void fragment() {
    // 计算当前 fragment 在 tile 网格中的坐标
    vec2 cell_uv = (UV * mask_size + map_offset) / mask_size;
    float visibility = texture(visibility_mask, cell_uv).a;
    
    // 根据 visibility 调整颜色
    vec4 tex_color = texture(TEXTURE, UV);
    if (visibility < 0.1) {
        // 未探索：暗色
        COLOR = vec4(0.02, 0.02, 0.05, 1.0);
    } else if (visibility < 0.6) {
        // 已探索但不在视野：混合暗色
        COLOR = mix(vec4(0.02, 0.02, 0.05, 1.0), tex_color, visibility);
    } else {
        // 在视野内：正常颜色
        COLOR = tex_color;
    }
}
```

#### CPU 侧提供 mask

与方案 C 相同的 Image 管理逻辑，区别在于 ImageTexture 不直接显示，而是传给 ShaderMaterial 的 uniform。

#### 对本项目的适用性

优点：
- GPU 计算，大规模地图性能更好
- 可以实现每个 tile 边缘柔和的迷雾过渡

缺点：
- 需要 shader 知识（本项目团队主要由策划/美术构成）
- `canvas_item` shader 在 TileMap 上的表现需要调试
- 本项目当前 TileMap 的 tile source 是纯色 ImageTexture，shader 叠加后视觉效果需要验证

结论：可作为未来优化方向，但不建议首版采用。


### 方案 E：独立迷雾 TileMapLayer（纯 GDScript 方案）

社区中最简单的实现：在原先的 TileMap 上再加一个 TileMapLayer，专用于迷雾覆盖。

#### 核心原理

1. 创建第二个 TileMapLayer（z_index 高于地形层）
2. 用纯黑半透明 tile 填充整个地图
3. 玩家移动时调用 `fog_layer.erase_cell(Vector2i(x, y))` 擦除可见区域的 tile

```gdscript
# 初始化：全图覆盖迷雾 tile
for x in range(map_width_tiles):
    for y in range(map_height_tiles):
        fog_layer.set_cell(Vector2i(x, y), FOG_SOURCE_ID, Vector2i(0, 0))

# 玩家移动时：圆形范围擦除
func reveal(player_pos: Vector2, radius: int) -> void:
    var cx := int(player_pos.x / TILE_SIZE)
    var cy := int(player_pos.y / TILE_SIZE)
    for dx in range(-radius, radius + 1):
        for dy in range(-radius, radius + 1):
            if dx * dx + dy * dy > radius * radius:
                continue
            fog_layer.erase_cell(Vector2i(cx + dx, cy + dy))
```

#### 性能分析

| 地图规模 | 迷雾 tile 数 | set_cell/erase_cell 开销 |
|----------|-------------|------------------------|
| 120x90 (本项目) | 10,800 | 初始化时 10,800 次 set_cell，可接受 |
| 每帧擦除 | ~80 格 (radius=5) | 几乎无开销 |

#### 对本项目的适用性

优点：
- 最简单，零 shader，纯 GDScript
- 与本项目 TileMap + ColorRect 体系天然兼容
- 迷雾 tile 可以用纯色 Image 生成（与现有 tile source 构建方式一致）

缺点：
- 视觉效果是 tile 级别的棋盘状，不够平滑
- 擦除的 tile 边缘锯齿感明显
- 无法实现 "已探索但不在视野" 的半透效果（要么有 tile 要么没有）


### 方案 F：Screen-Space Circle Reveal Shader

godotshaders.com 上的 Basic Fog of War Shader，用于简单圆形范围揭露。

#### 核心原理

在覆盖整个屏幕的 ColorRect 上挂载 shader，shader 读取 `SCREEN_TEXTURE` 并根据一组 reveal center 的距离决定显示/隐藏。

```glsl
shader_type canvas_item;
uniform sampler2D SCREEN_TEXTURE : hint_screen_texture;
uniform vec2 reveal_points[16];       // 最多 16 个 reveal 中心
uniform float reveal_radius = 150.0;

void fragment() {
    vec4 screen_color = texture(SCREEN_TEXTURE, SCREEN_UV);
    float min_dist = reveal_radius;
    
    for (int i = 0; i < 16; i++) {
        float dist = length(SCREEN_UV * screen_size - reveal_points[i]);
        min_dist = min(min_dist, dist);
    }
    
    if (min_dist < reveal_radius) {
        COLOR = screen_color;
    } else {
        COLOR = vec4(0.02, 0.02, 0.05, 1.0);
    }
}
```

#### 局限性

| 问题 | 影响 |
|------|------|
| reveal_points 数组大小固定 | 16 个点不够覆盖同屏多个实体 |
| 屏幕空间计算 | 摄像头移动时需要转换坐标 |
| 只支持圆形 reveal | 不支持任意形状的探索区域 |
| 像素级（非 tile 级） | 与 TileMap 的网格结构不匹配 |

对本项目不推荐。适合小场景、少量 reveal source 的项目。

## 四、综合推荐（2 个方案）

### 推荐方案 1：SubViewport 小地图 + ImageTexture 迷雾（首选）

将 SubViewport 小地图（方案 A）与 ImageTexture 迷雾（方案 C）组合：

**小地图层**（方案 A）：
```
CanvasLayer
└── MiniMapPanel
    └── SubViewportContainer (stretch)
        ├── SubViewport (共享 world_2d)
        │   ├── MinimapCamera (zoom≈0.08, 跟踪玩家)
        │   └── PlayerMarker (ColorRect 或 Sprite2D 纯色方块)
        └── (贴靠右上角, 约 180x180 像素区域)
```

**迷雾层**（方案 C，在现有 MapSystem 基础上重构）：
```
SubViewport 内部或主世界 CanvasLayer
└── FogOverlay (TextureRect, 覆盖整个 TileMap)
    texture = ImageTexture (实时更新)
    texture_filter = Nearest
    expand_mode = Ignore Size
    stretch_mode = Tile
```

#### 实施路径

| 步骤 | 内容 | 预估工时 |
|------|------|---------|
| 1 | 重构 MapPanel，引入 SubViewport + MinimapCamera | 2h |
| 2 | SubViewport 设置 `world_2d` 共享 | 5min |
| 3 | MinimapCamera zoom 调试（5760x4320 地图适配） | 15min |
| 4 | 将现有 FogOverlay Image 逻辑改为重用 Image/ImageTexture | 1h |
| 5 | 实现三层迷雾（未探索/已探索/视野内） | 1h |
| 6 | 移除旧的 MapPanel Label 标记，改用 SubViewport 内 Sprite2D | 30min |
| 7 | 测试 + 调优性能 | 1h |

#### 关键改造点

当前 `MapSystem._draw_fog()` 的问题（需改为）：
```gdscript
# 当前（每帧重建 Image，浪费）：
var img := Image.create(cols, rows, false, Image.FORMAT_RGBA8)
for x in cols: for y in rows: img.set_pixel(...)  # GDScript 循环 ~10,800 次/帧
var tex := ImageTexture.create_from_image(img)      # 每帧分配新 GPU 纹理！
fog.texture = tex

# 改为（初始化一次，增量更新）：
# _ready 时：
_fog_image = Image.create(cols, rows, false, Image.FORMAT_RGBA8)
_fog_image.fill(Color(0, 0, 0, 0.7))   # 全黑=未探索
_fog_texture = ImageTexture.create_from_image(_fog_image)
fog.texture = _fog_texture

# 每帧仅更新变化区域（~80 像素）：
for each newly explored cell (x, y):
    _fog_image.set_pixel(x, y, Color(0, 0, 0, 0.0))
_fog_texture.update(_fog_image)  # 增量 GPU 上传
```

性能提升：每帧 GDScript 循环从 10,800 次降低到 ~80 次（~135x 加速），GPU 纹理分配从"每帧新建"变为"增量更新已有纹理"。


### 推荐方案 2：ImageTexture 迷雾 + 轻量 Map Panel 改进（保守方案）

如果时间紧张不想重构 SubViewport，保持现有架构，仅做性能修复和质量提升。

**不做**：SubViewport 小地图重构  
**只做**：迷雾引擎性能优化 + 三层迷雾状态

#### 实施路径

| 步骤 | 内容 | 预估工时 |
|------|------|---------|
| 1 | 修复 `_draw_fog()` 为增量更新模式 | 30min |
| 2 | 实现三层迷雾状态：未探索(黑) → 已探索(灰半透) → 当前视野(透明) | 45min |
| 3 | 扩大玩家迷雾清除半径（当前 80px→建议 160px） | 5min |
| 4 | MapPanel 分辨率提升（当前 800x400→建议 1200x600） | 10min |
| 5 | 敌人仅在"当前视野内"时渲染（用 visibility layer 或 process_mode 控制） | 1h |

#### 敌人隐藏逻辑（配合迷雾）

```gdscript
# 在 enemy.gd 或 game_manager.gd 中
func _process(_delta: float) -> void:
    for enemy in active_enemies:
        var cell_x := int(enemy.global_position.x / TILE_SIZE)
        var cell_y := int(enemy.global_position.y / TILE_SIZE)
        var in_sight := _fog_grid[cell_x][cell_y] and _is_adjacent_to_player(cell_x, cell_y)
        enemy.visible = in_sight
        enemy.set_process(in_sight)
```

## 五、不推荐的方案及原因

| 方案 | 不推荐原因 |
|------|-----------|
| Screen-Space Shader（方案 F） | 固定数组大小、屏幕空间计算复杂、不适合俯视角大地图 |
| 独立 TileMapLayer（方案 E） | 无法做"已探索但不在视野"的半透明效果，视觉粗糙 |
| Light2D + Occluder（Darkwood 风格） | 需要给所有墙壁加 Occluder2D，本项目 120x90 大地图不可行 |
| 引入 Asset Library 插件 | 约束明确禁止，且 Mini Map 插件的功能用原生 SubViewport 即可实现 |

## 七、最终建议

**首选方案 1**（SubViewport 小地图 + ImageTexture 迷雾）。理由：

1. SubViewport 是 Godot 内置功能，不引入任何外部依赖
2. `world_2d` 共享机制意味着零节点复制、零同步逻辑
3. ImageTexture 增量更新是本项目已有方向，只需做性能修复
4. 两者组合可覆盖"小地图显示 + 全域战争迷雾"的完整需求
5. 与本项目 ColorRect 纯色体系完全兼容
6. 预估总工时 5-6 小时，风险低

如果资源紧张，退而求其次用**方案 2**（仅迷雾优化），但它缺失了小地图的实时动态显示能力。
