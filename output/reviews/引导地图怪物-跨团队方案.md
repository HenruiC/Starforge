# 紧急跨团队方案：引导 + 地图 + 怪物技能

## 小岛秀夫开场白

我闭上眼睛，想象一个玩家——第一次进入这个校园。她杀了几个怪物。屏幕上闪过"清理教室区的怪物 (15)"。她去了一间教室，杀了一些。又去了另一间。杀完了。然后……什么也没有发生。没有方向。没有声音。没有线索。她开始在空荡荡的走廊里游荡，五分钟后退出了游戏。

这不是一个坏系统。这是一个系统还没有被完成。

我们有三件事需要同时修复。每一件都不是"加点代码"——每一个都是"完成那个还没被写完的系统"。

## 议题1：玩家引导 —— 不只是"箭头指向哪"

### 诊断

**根因不是坐标错了，是 MissionPromptUI 是一具空壳。**

`scripts/ui/mission_prompt_ui.gd` 的每一个方法都是 `pass`。它不渲染任何东西。不显示标题。不显示进度。不画箭头。

而 `game_manager.gd` 第 1051-1062 行确实调用了 `_mission_prompt_ui.show_direction_arrow(Vector2(...))` ——但这个方法根本不存在于 MissionPromptUI 类中。`has_method` 返回 false，静默跳过。

**第二个问题：Stage 2 的箭头目标写错了。**
`mission_manager.gd` 第 194 行：`prompt2_on.arrow_target_zone = "zone_gym_entrance"`。玩家在 Stage 2 需要在教室里杀敌，箭头却指向体育馆——这是误导。

**第三个问题：Stage 2→3 过渡时没有激活中提示。**
Stage 2 完成后，Stage 3 因 `activation_condition`（`LOCATION_REACH zone_gym_entrance`）处于 PENDING 状态。此时没有任何 UI 告诉玩家"体育馆入口在等你"。

### 方案

**修复1：让 MissionPromptUI 真正渲染（交互策划 + 马斯克）**

MissionPromptUI 需要实现三个核心渲染：

```
A. 顶部标题条：当前阶段名称，从屏幕顶部滑入，持续显示
B. 方向箭头：一个三角形指示器，指向目标坐标/Zone（考虑屏幕边缘clamp）
C. 目标进度条/计数器：显示 kill_count 或其他 Objective 进度
```

具体实施：
- `show_stage_activate(title_text)` → 实例化 TitleLabel + 入场动画（从顶部 slide_down）
- `show_direction_arrow(target: Vector2)` → 计算屏幕方向向量，在屏幕边缘 clamp 放置箭头 TextureRect
- `show_objective_progress(...)` → 底部横向进度条
- `complete_objective(...)` → 绿色对勾标记 + 短暂闪烁
- `dismiss_prompt(prompt_id)` → 淡出

**修复2：修正 Stage 2 的箭头目标**

`mission_manager.gd` 第 194 行，将 `arrow_target_zone` 从 `"zone_gym_entrance"` 改为教室区域坐标。简单方案：Stage 2 不显示方向箭头，改为文本提示"探索教学楼两侧教室"。

如果坚持保留箭头，需要新增一个 zone：`zone_classroom_cluster`。但这引入不必要的复杂度。我建议 Stage 2 用文字描述代替箭头——教室是一个区域，不是一个点。箭头指向一个"区域中心"会让玩家感到困惑。

**修复3：增加 Stage 间过渡提示（系统策划 + 陶德）**

当 Stage 完成且有 PENDING 的下一 Stage 时，触发一个"过渡提示"：

```
文字："体育馆深处传来一声哨响..."
箭头：指向 zone_gym_entrance
持续时间：直到玩家进入 zone_gym_entrance
```

这需要在 MissionTriggerManager 中新增一个信号 `stage_pending(next_stage_id: int, target_zone_id: String)`，由 GameManager 监听并驱动 UI。

**修复4：Boss 间入口视觉标识（宫崎英高 + 关卡策划）**

体育馆入口目前没有任何视觉提示。方案：
- 在入口上方放置一个红色发光标记（类似 Door 节点但颜色是红色）
- 进入该区域时播放低沉的环境音（AudioStreamPlayer 预加载）
- 可选：在地面上放置一排逐渐变暗的血迹引导线

## 议题2：怪物技能 —— 技能系统建好了，没人给怪物用

### 诊断

**根因：enemy.gd 完全绕过了 AIBehaviorController 和 SkillBase 体系。**

当前敌方行为执行路径：
```
Enemy._physics_process()
  → _melee_behavior() 或 _ranged_behavior()
    → 直接设置 velocity，接触伤害靠 hitbox 碰撞
    → _shoot() 直接实例化 bullet
```

但正确的执行路径应该是：
```
Enemy._physics_process()
  → AIBehaviorController._process()
    → _tick_chase() 或 _tick_kite()
      → _try_melee_attack() 调用 skill.try_execute()
        → SkillBase 协程：前摇 → 伤害帧 → 硬直
```

**两个问题合为一个：**
1. enemy.gd 没有创建 AIBehaviorController 实例
2. enemy.gd 没有分配技能给敌人

同时，闪烁问题的根因是：enemy.gd 直接操作 `sprite.modulate`（第 133-134 行设为 GREEN，第 156 行设为 WHITE），而 visual_state_machine 也通过 `_apply_color_tween` 修改同一个 modulate 属性。两套系统在争抢同一个属性。

### 方案

**修复1：给 Enemy 挂载 AIBehaviorController + 技能（宫崎英高 + 马斯克）**

在 enemy.gd 的 `_ready()` 中：
```gdscript
# 创建 AI 控制器
var ai_ctrl := AIBehaviorController.new()
ai_ctrl.name = "AIBehaviorController"
add_child(ai_ctrl)
ai_ctrl.setup(self, _player_ref)

# 根据敌人类型分配技能
var skills: Array = []
if is_melee and not is_elite:
    skills.append(_create_melee_skill("slash"))
    skills.append(_create_melee_skill("dash"))  # 新增：冲刺技能
elif is_ranged:
    skills.append(_create_ranged_skill("multi_shot"))
elif is_elite:
    skills.append(_create_melee_skill("whirlwind"))
    skills.append(_create_melee_skill("dash"))

# 挂载技能到控制器
ai_ctrl.connect_skills(skills)

# 替换 _physics_process，改为调用 ai_ctrl 的 tick
```

**修复2：为敌人创建专用技能子类（宫崎英高）**

不需要从头写。SkillBase 体系已经完备，只需要创建三个敌人专用技能：

| 技能ID | 类型 | attack_category | 前摇 | 冷却 | 效果 |
|--------|------|-----------------|------|------|------|
| `enemy_slash` | 近战 | melee_light | 0.35s | 1.0s | 前方弧形斩击，damage=15 |
| `enemy_dash` | 近战 | dash | 0.6s | 3.0s | 快速冲刺+冲撞伤害 |
| `enemy_aoe` | 精英 | aoe_ground | 0.8s | 5.0s | 圆形冲击波 |

实现方式：直接复用 `SkillSlash` 的模式——继承 SkillBase，覆盖 `execute()`。`enemy_dash` 在 execute 中设置 velocity 突进。

**修复3：解决闪烁冲突（马斯克）**

两种方案选一：
- **方案A（推荐 —— 快）：** 删除 enemy.gd 中所有直接操作 `sprite.modulate` 的代码（`_show_hit_effect` 中的 WHITE、`_shoot` 中的 GREEN），改为通过 VisualStateMachine 接口。受击时调用 `visual_state_machine._on_hit()`，射击时暂不处理闪光。
- **方案B（彻底 —— 慢）：** 完全移除 enemy.gd 中的 `_setup_elite`、`_setup_boss` 中的颜色/scale 操作，全部委托给 EnemyVisualFactory + VisualStateMachine。

推荐方案A，1小时内完成。方案B作为 Phase 5 重构任务。

## 议题3：地图系统 —— 纹理生成了，但没贴上

### 诊断

MapPanel 空白的可能原因有三个（需要按顺序排查）：

**原因1：MapSystem.init() 时机问题。**
`game_manager.gd` 第 369-375 行在 `_on_game_start()` 中调用 `MapSystem.init()`。但 SchoolMap 的 TileMap 在 `_ready()` 中通过 `call_deferred("_deferred_update")` 异步构建。如果 `_on_game_start` 在 TileMap 完成前执行，`generate_map_texture()` 返回的纹理是空的（所有地板都是默认色）。

**原因2：MapPanel 节点路径不匹配。**
`map_system.gd` 第 12 行硬编码 `mp.get_node("MapTexture")`。如果场景树中这个路径不存在，会抛异常。需要确认 `MapPanel` 内部确实有名为 `MapTexture` 的 TextureRect 子节点。

**原因3：雾层覆盖了整个地图。**
`map_system.gd` 的 `_draw_fog()` 用 fog_grid 尺寸（约 67x50 像素）创建 Image，拉伸到全尺寸。如果 fog 初始化失败，整个雾层可能是不透明的黑色/深蓝色，盖住了地图纹理。

**原因4（最可能）：`MapSystem.update()` 没有被持续调用。**
`game_manager.gd` 只在 `_toggle_map()` 中调用 `_update_map_from_system()`（第 1092 行），这更新了玩家标记。但在 `_process` 中没有每帧调用。这一点其实对——地图打开时才需要更新位置。但 init 时的纹理设置可能已经失败。

### 方案

**修复1：确保 generate_map_texture 在 ImageTexture 创建前完成（马斯克）**

最简单方案：不依赖 SchoolMap 的 TileMap。在 `_on_game_start` 中直接调用 `generate_map_texture()` 之前，如果 TileMap 未就绪，加一个 await：

```gdscript
# 等待 SchoolMap 的 TileMap 完成
var sm := get_node_or_null("../SchoolMap")
if sm and sm.has_method("generate_map_texture"):
    if sm.get_node_or_null("TileMap"):
        await get_tree().process_frame  # 等一帧
        map_tex = sm.generate_map_texture()
```

**修复2：地图上标注关键点（交互策划）**

地图纹理生成后，在 MapSystem 中增加 POI 标注层：
- 玩家位置（已有 PlayerMarker，需确认可见）
- 校门（金色标记，坐标约 1600, 2320）
- 体育馆入口（红色标记，坐标约 1584, 600）
- Boss 间（红色警告标记，坐标约 1584, 500）
- 当前任务目标区域（用 pulsing 光圈标记）

POI 用简单的 TextureRect 或 Label 放在 MapPanel 上，位置用地图坐标到面板坐标的映射计算。

**修复3：雾层渲染优化（马斯克）**

当前雾层的 Image 分辨率是 67x50 像素，拉伸到大约 800x400 的面板上会非常模糊。改为：
- 用一个半透明黑色 TextureRect 覆盖整个 MapPanel
- 在玩家周围用 CanvasItem 绘制一个透明圆形/矩形区域
- 或者提升雾层 Image 分辨率到至少 200x150，用 `Image.EXPAND_NEAREST` 避免模糊

## 议题4：Boss 战 —— 能生成，但 Boss 没有脑子

### 诊断

**根因：Boss 敌人创建后没有挂载 BossAI 或 BossPhaseController。**

`game_manager.gd` `_on_player_entered_zone_for_boss`（第 910-973 行）创建 Boss 敌人的流程：
```gdscript
var e: CharacterBody2D = _enemy_scene.instantiate()
e.is_boss = true
enemies.add_child(e)
```

问题：`e.is_boss = true` 只触发了 `_setup_boss()`（改变颜色/大小/光环），但 **没有创建 BossAI 控制器，没有创建 BossPhaseController，没有分配 Boss 技能**。

所以这个 Boss 的行为就是——一个大了三倍、红色的近战杂兵。它会走向玩家，接触造成伤害。不会放技能，不会转阶段，不会召唤学生。什么都没有。

其次，zone 坐标需要验证。zone 定义中的 `zone_gym_entrance` 位置 (1584, 600) 和 `zone_gym_boss` 位置 (1584, 500) 必须与实际地图布局对齐。两个地图文件（`school_map.gd` 和 `map_school.gd`）的坐标系统不完全一致，需要确认当前用的是哪个。

### 方案

**修复1：Boss 生成时挂载完整 AI + Phase 系统（马斯克 + 宫崎英高）**

在 `_on_player_entered_zone_for_boss` 中，创建 Boss 后立即初始化 Boss 专属组件：

```gdscript
# 创建 BossPhaseController
var phase_ctrl := BossPhaseController.new()
phase_ctrl.name = "BossPhaseController"
phase_ctrl.setup(e, _dungeon_config)
e.add_child(phase_ctrl)

# 创建 BossAI
var boss_ai := BossAI.new()
boss_ai.name = "BossAI"
boss_ai.setup(e, player, phase_ctrl, boss_skills)
e.add_child(boss_ai)

# 创建 VisualStateMachine（如果尚未由工厂创建）
# 连接 boss_ai.attack_phase_changed → visual_state_machine
```

但这里有一个架构冲突：enemy.gd 的 `_physics_process` 直接调用 `_melee_behavior()`，会与 BossAI 的 `tick()` 产生双重移动。解决方案：
- Boss 敌人进入时设置一个标志 `_use_boss_ai = true`
- 在 `_physics_process` 开头检查此标志，跳过原有逻辑
- 改为在 `_process` 中调用 `boss_ai.tick(delta)`

**修复2：在 BossHPBar 中连接血量更新（马斯克）**

`_on_player_entered_zone_for_boss` 已经创建了 BossHPBar 并调用 `enter()` 和 `set_hp()`。但 Boss 受到伤害后，血条需要持续更新。需要在 Boss 的 `damage_taken` 信号中更新血条：

```gdscript
e.damage_taken.connect(func(_, current_hp, _):
    if _boss_hp_bar and is_instance_valid(_boss_hp_bar):
        _boss_hp_bar.set_hp(current_hp, e.max_health)
)
```

**修复3：Zone 坐标实地验证（关卡策划）**

在编辑器中可视化 Zone 区域（临时绘制矩形），确认：
- `zone_gate`: (1600, 2320) 是否在校门区域
- `zone_gym_entrance`: (1584, 600) 是否在体育馆入口
- `zone_gym_boss`: (1584, 500) 是否在体育馆内部

如果不匹配，更新 `mission_manager.gd` 中 `_create_school_chapter()` 的坐标。

## 最终裁决：Strand / Fire

这四个问题有共通的本质：**系统架构被拆成了多个文件，但文件之间的连接线没有被完成。** 这不是缺失功能，是被遗忘的胶水代码。

MissionPromptUI 是空壳 ←→ game_manager 调用了不存在的方法
SkillBase 体系完备 ←→ enemy 从未被分配技能
BossAI 实现完整 ←→ Boss 生成时没有挂载
地图纹理生成存在 ←→ 但不知道何时/是否成功传递到 MapSystem

修复不需要重新设计。修复是连接已经存在的点。

**裁决：Strand**

不是重新点燃（Fire），是把断掉的线重新接上（Strand）。Kojima Productions 的标志是绳——连接。我们现在做的事情，就是让系统之间重新连接。

每个任务单独来看都很小（1-2小时），但累积起来就是玩家今晚能否玩到 Boss 战的区别。
