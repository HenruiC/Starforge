### P0-4: Player.take_damage() 绕过 CombatUnit 防御计算链

**文件**: `scripts/player.gd:91-104`

```gdscript
func take_damage(amount: int, source: CombatUnit = null) -> void:
    if is_dead:
        return
    var actual: int = max(amount - defense, 1)  # 直接用 @export var defense
    _health = max(_health - actual, 0)
    # ... 不调用 super.take_damage()
```

CombatUnit.take_damage() 使用 `get_stat("defense")` 通过 StatsComponent 读取，受 Buff/StatModifier 影响。Player 绕过它直接用 `defense` 属性。当 StatsComponent 挂载后，Buff 对防御的修改对 Player 无效。

**修复**: 两种方案：
- **方案A**: 在 `take_damage` 开头调用 `super.take_damage(amount, source)`，然后仅保留 UI 更新和 hit flash
- **方案B**: 用 `get_stat("defense")` 替换 `defense`，保持与 CombatUnit 计算一致

### P1-4: AIBehaviorController 的 _should_stun() 逻辑死循环

**文件**: `scripts/ai/ai_behavior_controller.gd:200-202`

```gdscript
func _should_stun() -> bool:
    return _current_state == AIState.STUNNED
```

此函数检查"是否应该进入 STUNNED 状态"，但它的逻辑是"如果当前已是 STUNNED 则保持 STUNNED"。这导致：一旦进入 STUNNED，`_evaluate_transition()` 永远不会跳出 STUNNED（因为 `_should_flee()` 不会被评估——STUNNED 优先级最高且永远为 true）。

**修复**: 应该由外部（BuffComponent）通过 `_unit.set_stunned(true)` 触发，然后 `_should_stun()` 检查外部 stun 标志：
```gdscript
func _should_stun() -> bool:
    if _unit.has_method("is_stunned"):
        return _unit.is_stunned()
    return false
```

### P1-8: _on_boss_health_changed 中条件检查有误

**文件**: `scripts/enemy.gd:316-319`

```gdscript
func _on_boss_health_changed(_current_hp: int, _max_hp: int) -> void:
    if _phase_controller and is_instance_valid(_phase_controller):
        var ratio: float = float(_current_hp) / float(_max_hp) if _max_hp > 0 else 0.0
```

|------|-------------------|-------------------|------|
| `_ready()` | 添加 `_build_low_hp_overlay()` 和 `_build_mission_prompt_ui()` | 添加 `_create_boss_ui_controllers()` | Phase C 的 `_build_low_hp_overlay()` 内部调用 `_build_mission_prompt_ui()`，与 Phase D 的 `.connect` 调用同时存在——无直接冲突，但 `_build_mission_prompt_ui()` 被调用两次（见 P1-3） |
| `_process()` | 添加 `_update_low_hp_overlay()` | 无修改 | 无冲突 |
| `_on_player_died()` | 大幅重写 GameOver 序列 | 无修改 | 无冲突 |
| `_show_upgrade_panel()` | 添加 Operator Protocol (5%概率) | 添加 Boss 战后 Operator Protocol (15%概率) + "佐藤的馈赠" | Phase C 和 D 对同一函数做了叠加修改，逻辑正确但 `_is_post_boss_upgrade` 标志在 Phase D 设置后在 `_enter_upgrade_panel()` 中消费——需确保每次升级后正确重置 |

### 冲突 C-2: enemy.gd — AI 系统和 Boss 系统共存

Phase B (AI 系统) 在 `_ready()` 中创建 `AIBehaviorController` 和 `EnemyVisualStateMachine`。
Phase E (Boss 系统) 在 `_setup_boss()` 中销毁 AIBehaviorController 并替换为 `BossAI`。

**风险**: 如果 `is_boss` 标志在 `_ready()` 之后才设置（例如通过代码设置 `enemy.is_boss = true`），则不会调用 `_setup_boss()`，Boss 仍使用标准 AI。

实际上 spawn 代码在 `game_manager.gd:743-749` 中设置 `e.is_boss = true` 在 `enemies.add_child(e)` 之前。在 Godot 中，`add_child` 触发 `_ready()`，所以 `_ready()` 执行时 `is_boss` 已为 true。这是安全的。
