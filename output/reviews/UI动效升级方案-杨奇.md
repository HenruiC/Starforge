# UI 动效升级方案

> **审定人：杨奇** | 2026.05.18
> 项目：combat-demo (Godot 4.6 幸存者Like战斗原型)
> 方案性质：美术方向指导 — 不含代码实现

---

## 一、现状诊断：一张没有呼吸的画

我通读了当前 UI 的全部实现。说一句实话：**骨骼都在，血肉全无。**

DamageNumber 的浮动 Tween 是唯一会呼吸的东西。CombatFeedback 的击杀粒子爆发也能看。但除此之外，所有 UI 面板的切换都是生硬的 `visible = true/false`，所有按钮都是不会回应的冰冷方块，所有数值变化都是文本的直接替换。

我们的项目有六个面板（CharSelect / HUD / LevelUp / GameOver / Map / Dialogue），每一个都是一个重要的叙事节拍。角色选择是"命运的第一次抉择"，升级面板是"成长的关键节点"，死亡面板是"旅程的句号"。现在它们给人的感觉，却是同一套面无表情的开关逻辑。

xcount 那套视频讲的东西，核心不是技法本身——是**让 UI 成为一个会呼吸的生命体**。按这个标准审视我们当前的代码，缺口明确，方向清晰。

## 二、技法映射：五把刀的落点

### 技法一：按钮悬停缩放+旋转

**适用面板：全部。**

当前我们通过 `_mk_btn()` 和 `_mk_style()` 用纯代码生成了所有按钮。StyleBoxFlat 只区分了 `normal` 和 `hover` 两种静态样式——换了个背景色而已。这是一个最低限度的交互反馈。

**具体落点：**

| 面板 | 控件 | 悬停方案 |
|------|------|----------|
| CharSelect | 9 个天赋按钮 (175×42) | 轻微缩放 1.03~1.05 + 边框色从 `(c.r, c.g, c.b, 0.3)` 过渡到 `#c9a84c` 金色辉光。**注意：长条形按钮旋转角应 < 1.5°，位移感知控制在 2px 以内** — 宽高比越大，旋转角越小 |
| CharSelect | 3 个武器按钮 (170×55) | 同上。武器按钮更偏方形，旋转角可以略大（~2°），配合 scale 1.05 |
| CharSelect | "踏入试炼"按钮 (280×48) | 极长按钮。旋转角压缩到 0.8°，主要靠缩放 + 边框辉光来传达反馈。**长按钮旋转是重灾区——手臂太长，末端位移会被放大** |
| LevelUpPanel | 3 个升级选项 (180×80) | 相对方形的按钮，旋转角可到 2.5°，scale 1.06，配合边框从深色过渡到暗金 `(0.8, 0.6, 0.1, 0.8)`。这是游戏中最重要的三选一抉择，悬停反馈应该有一种"被选中"的庄重感 |
| GameOver | 目前只有文字提示"按 W 重新开始"，没有按钮控件 | 暂不适用。但考虑将来改为按钮时参考 CharSelect 方案 |

**视觉风格约束：** 悬停缩放时，Tween 时长控制在 0.12~0.18 秒，ease 用 `EASE_OUT`。不要做弹性（bounce）——暗色主题的交互反馈应该是克制的、有分量的。金色辉光的过渡比缩放本身更重要。

### 技法二：容器子元素依次出现（Stagger）

**这是当前面板体验的最大痛点。**

LevelUpPanel 的三个升级按钮、CharSelect 的武器区/天赋区/预览区、GameOver 的标题/分数/提示——它们同时出现，像一盏日光灯突然亮起。没有任何戏剧性。

**具体落点：**

| 面板 | 子元素序列 | 延迟方案 |
|------|-----------|----------|
| **LevelUpPanel** (最高优先级) | 1) 半透明黑底 → 2) "升级！"标题 → 3) 三个升级按钮 | 遮罩 0.15s 淡入 → 标题 0.1s 后从上方 20px 滑入(0.25s) → 三个按钮各间隔 0.08s 依次从下方升起(0.3s, EASE_OUT_BACK mild) |
| **CharSelect** | 1) 深蓝背景 → 2) 叙事引文 → 3) 武器区 → 4) 天赋区 → 5) 预览区 → 6) 确认按钮 | 背景 0.2s → 引文 0.3s 淡入（保持文字的尊严，不用位移，只有透明度）→ 三个区块各间隔 0.12s → 确认按钮最后 0.2s 从下方滑入 |
| **GameOver** | 1) 暗色遮罩 → 2) "你阵亡了"标题 → 3) 战绩数据 → 4) 重新开始提示 | 遮罩 0.3s（慢！死亡是重要的）→ 标题 0.5s 后从中心缩放弹出(1.0→1.05→1.0) → 战绩 0.2s 后淡入 → 提示 1.0s 后再出现（给玩家消化情绪的时间） |
| **DialoguePanel** | 1) 底部黑色条 → 2) 说话人名称 → 3) 对话文本 | 黑条从底部滑入 0.25s → 说话人 0.05s 后淡入 → 文本 0.1s 后开始逐字（见技法四补充说明） |
| **HUD 任务目标** | 多条 `□ 目标` 行 | 每条目标行间隔 0.06s 依次淡入。阶段切换时旧目标淡出、新目标依次淡入 |

**视觉风格约束：** 暗色主题下的 stagger 不要用明快的弹跳。用 `EASE_OUT` 缓出，让元素有一种"从黑暗中浮现"的感觉。位移幅度不要超过 20px——暗示而非张扬。

### 技法三：Auto-Tween 脚本（无侵入淡入淡出）

**最应该立刻做的架构级改进。**

当前 `PanelManager.show()` / `hide()` 是直接的 `visible = true/false`。所有面板的生死都在一瞬间。

**落点方案：**

创建一个全局可复用的 Tween 辅助（不写代码，只讲方向）：
- 为 `PanelManager` 的 `show()` / `hide()` 注入统一的淡入淡出
- `show()` : 先设置 `visible = true` + `modulate.a = 0.0`，Tween 0.25s 淡入到 `modulate.a = 1.0`
- `hide()` : Tween 0.2s 淡出到 `modulate.a = 0.0`，完成后设置 `visible = false`
- 对于需要特殊动效的面板（如 LevelUpPanel 有 stagger），保留自定义入口，不覆盖

**具体应用于：**

| 面板 | 现状态 | 升级后 |
|------|--------|--------|
| CharSelect | `visible = true` (瞬间) | 0.35s 整体淡入 + stagger 子元素（见技法二） |
| LevelUpPanel | `visible = true` + `get_tree().paused = true` (非常生硬) | 0.25s 遮罩淡入 + 子元素 stagger |
| MapPanel | M 键切换 `visible` | 0.2s 缩放淡入(scale 1.05→1.0 + fade)，关闭时反向。配合迷雾本身就是一种叙事节奏 |
| GameOver | `visible = true` 瞬间出现 | 0.5s 遮罩淡入 + 子元素 stagger |
| DialoguePanel | `visible = true` 瞬间 | 底部黑条 0.25s 滑入 + 文本出现（见下） |

**视觉风格约束：** 所有淡入淡出统一使用暗金琥珀 `#c9a84c` 作为过渡期间可能的强调色（比如边框光晕在淡入过程中短暂闪烁）。但不要滥用——只有 LevelUpPanel 和 CharSelect 这两个叙事关键面板配得上金色辉光过渡。HUD 和 Map 用纯粹的透明度过渡即可。

### 技法四：重写 Setter 让数值弹跳

**本项目的 HUD 存在严重的"数字尸体"问题。**

击杀数从 0 跳到 15 没有任何过渡。波次切换没有仪式感。计时器就是一个冷漠的递减数字。这些数字是玩家与世界的唯一量化连接。

**具体落点：**

| 控件 | 当前行为 | 弹跳方案 |
|------|----------|----------|
| **KillCount** `击杀: N` | `kill_label.text = "击杀: %d" % _kill_count` | 数值变化时：当前数字先 flash 放大(1.15x, 0.08s) → 缩小回 1.0x → 更新为新数字 → 新数字 flash 放大(1.2x, 0.1s) → 回弹 1.0x。关键击杀（精英/Boss）延迟 + 大 flash |
| **WaveLabel** `波次 N` | `wave_label.text = "波次 %d" % _current_wave` | 波次切换是最重要的节奏提示。文字 scale 从 1.0→1.3→1.0 (0.3s, EASE_OUT_BACK)，字体颜色短暂闪过金色再回到暗金 |
| **TimerLabel** `剩余: Ns` | `timer_label.text = "剩余: %.0fs" % ...` | 日常倒计时无需弹跳。但最后 10 秒：数字颜色从灰白过渡到绯红 `#c94a4a`，每秒数字变化时做轻微 pulse (1.0→1.08→1.0, 0.15s)。最后 3 秒：pulse 频率加倍，颜色变为 `#ff3020` |
| **FinalScore** 战绩数据 | `final_score_label.text = "击杀: %d\n..."` | GameOver 面板出现后，战绩数字不要一次性显示。做一个 0.5s 的计数动画：数字从 0 滚动到最终值（类似老虎机但比老虎机克制），滚动期间数字微微抖动 |
| **MissionObjectives** 进度数字 | `lines += "(%d/%d)" % [pv, tv]` | 进度增加时：完成的那一行短暂变为金色然后恢复。从 `□` 变为 `✓` 时，`✓` 做一个 0.15s 的描边发光 pulse |

**视觉风格约束：** 弹跳不是卖萌。暗色主题下的弹跳幅度要克制——scale 最大不要超过 1.3。Tween 时长 0.08~0.3 秒，最快的用于高频更新的击杀数，最慢的用于波次切换。弹跳的缓动用 `EASE_OUT_BACK`（轻微回弹），但 `overshoot` 参数控制在 0.3 以下。

### 技法五：根据控件尺寸动态调整动效强度

这个技法的核心思想是：**动画参数不应该写死，应该根据控件的物理尺寸来计算。**

**应用于本项目：**

| 场景 | 问题 | 动态方案 |
|------|------|----------|
| 按钮尺寸差异 | CharSelect 天赋按钮 175×42、武器按钮 170×55、确认按钮 280×48、升级按钮 180×80——如果都用固定旋转角度，长按钮的边角位移会很夸张 | 旋转角 = `base_rotation / (button_width / button_height)`。宽高比越大，旋转角越小。基础旋转角设 2.5°，对于宽高比 5.8 的确认按钮，实际旋转角约 0.4° |
| 不同面板的淡入时长 | 全屏大面板（CharSelect/MapPanel）和半屏面板（LevelUpPanel/GameOver）的淡入不应该用同一个时间常数 | 淡入时长 = `0.15 + (面板面积 / 屏幕面积) * 0.25`。大面板给更多时间呼吸 |
| 文字尺寸差异 | KillCount(font_size 22) 和 WaveLabel(font_size 16) 弹跳 scale 如果用同一个值，小字会被弹跳淹没 | 弹跳 scale = `1.0 + (24 / font_size) * 0.1`。小字跳得稍大一些，大字微跳即可 |
| stagger 间隔 | 3 个元素的容器和 6 个元素的容器不能用同一个 stagger 延迟 | 总 stagger 窗口固定 0.4~0.5s，每个元素的延迟 = `0.5s / 元素数量`。保证整体节奏紧凑 |

**视觉风格约束：** 动态计算的核心原则——**大控件微动，小控件显动**。这不是技术问题，是视觉重量感的问题。一个巨大的面板如果跳得太猛，会让人觉得 UI 要散架了。

## 三、视觉风格的统一基调

xcount 的视频讲的是"术"，下面这段讲的是"道"——所有动效必须服从于项目的美术基调。

### 暗而不黑：暗色体系的色彩动效

我们的背景是 `Color(0.06, 0.06, 0.12, 1)`——一种有蓝色底调的深空。这不是一片死黑，而是夜空。所有动效的颜色必须从这个基底中生长出来：

- **淡入/淡出** 不使用纯黑遮罩，使用 `Color(0.02, 0.02, 0.06, 0.92)`——与 CharSelect 的背景一致，带一点蓝底
- **高亮/悬停** 使用暗金琥珀 `#c9a84c`（R:0.79, G:0.66, B:0.30）及其低透明度变体。金色是这个世界的"希望色"
- **危险/警告**（倒计时最后 10 秒）使用绯红 `#c94a4a`，但不要用纯红 `#ff0000`——太刺眼了，暗色主题承受不了那个亮度
- **确认/完成** 使用极淡的翠绿色调 `#4a8c5c`，不要荧光绿

### 留白与呼吸：动效的节奏

每个动效结束后，必须有 0.1~0.3 秒的静默期（"hold"），然后才进入下一个动效。不要一个动效紧接另一个——"无动画更好"永远优先于"加更多动画"。

**三条铁律：**
1. 同一个 0.5 秒窗口内，最多只能有一个面板在做动效
2. 战斗中的 HUD 动效（击杀数弹跳）绝对不能干扰玩家的注意力——pulse 幅度必须低于 1.15
3. 暂停状态下（LevelUpPanel / CharSelect）的动效可以更大胆——玩家的大脑此时有带宽

### 质感优先：我们的 Tween 要有重量

不要用线性过渡（linear tween），那东西没有质感。我们的世界是粗糙的、有分量的：
- 所有位移动效用 `EASE_OUT`（减速入场，有质量感）
- 所有淡入动效用 `EASE_IN_OUT`（缓缓浮现）
- 弹跳用 `EASE_OUT_BACK` 但 overshoot 控制在 0.2~0.3（微回弹，不油腻）

## 四、优先级排序

### 第一梯队：核心体验修复（立即启动）

这些是 **不做的代价远比做要高** 的东西——当前玩家每分每秒都在忍受生硬的 UI。

| 序号 | 事项 | 涉及文件/面板 | 预估复杂度 | 理由 |
|------|------|-------------|-----------|------|
| **P0-1** | PanelManager 统一淡入淡出 | `panel_manager.gd` → 所有面板的 show/hide | 低 | 一个改动覆盖所有面板。这是地基。不修这个，后面所有面板级动效都要各自为战 |
| **P0-2** | LevelUpPanel stagger 出场 | `game_manager.gd` `_show_upgrade_panel()` | 中 | 这是玩家看到最频繁的暂停 UI。三个按钮同时蹦出来，玩家甚至来不及意识到"需要做选择了"。stagger 不仅好看，更有功能价值——引导视觉注意 |
| **P0-3** | KillCount 数字弹跳 | `game_manager.gd` `_update_ui()` + KillCount Label | 低 | 核心循环反馈。击杀数没有动效，等于告诉玩家"你做的事情不重要" |

### 第二梯队：品质感知提升（本周内）

这些东西决定了玩家对游戏的第一印象和最后印象。

| 序号 | 事项 | 涉及文件/面板 | 预估复杂度 | 理由 |
|------|------|-------------|-----------|------|
| **P1-1** | 全部按钮悬停动效 | `game_manager.gd` `_mk_btn()` | 中 | 按钮是玩家与 UI 交互的唯一触点。悬停没有动效，像在用一个不会回应的工具。注意长按钮的动态旋转角 |
| **P1-2** | CharSelect stagger 出场 | `game_manager.gd` `_show_char_select()` | 中 | 这是玩家进入游戏看到的第一个画面。叙事引文逐行淡入 → 三个选择区依次出现 → 确认按钮滑入。要有"翻开一本旧书"的仪式感 |
| **P1-3** | GameOver 序列出场 | `game_manager.gd` `_on_player_died()` | 中 | 死亡是叙事高潮后的留白。面板不应该瞬间弹出来。慢一点，给玩家看战绩、消化情绪的时间 |
| **P1-4** | HUD WaveLabel 波次切换动画 | `game_manager.gd` `_on_wave_timer_timeout()` | 低 | 波次切换是游戏节奏的鼓点。现在就是改个数字——没有鼓点。金色 flash + scale pulse |

### 第三梯队：细节打磨（迭代中逐步加入）

这些是有价值的，但不阻碍核心体验。可以随着迭代逐步添加。

| 序号 | 事项 | 涉及文件/面板 | 预估复杂度 | 理由 |
|------|------|-------------|-----------|------|
| **P2-1** | TimerLabel 倒计时紧迫感 | `game_manager.gd` `_process()` | 低-中 | 最后 10 秒颜色渐变 + pulse。工期小，但只影响体验的最后一小段 |
| **P2-2** | DialoguePanel 文本逐字出现 + 滑入 | `dialogue_panel.gd` / `dialogue_panel.tscn` | 中 | 对话面板现在是一整段文本瞬间出现——完全没有阅读节奏。逐字出现(30字符/秒) + 底部黑条从下方滑入 |
| **P2-3** | MapPanel 开启动效 | `game_manager.gd` `_input()` M键处理 | 低 | 打开地图时缩放淡入(scale 1.05→1.0 + fade 0.2s)，关闭时反向。配合迷雾展开的渐进揭示感 |
| **P2-4** | MissionObjectives 目标完成动画 | `game_manager.gd` `_update_mission_hud()` | 中 | `□→✓` 的短暂金色 pulse。进度条类目标的进度数字弹跳 |
| **P2-5** | CharSelect 天赋选中动效 | `game_manager.gd` `_toggle_talent()` | 低 | 选中/取消天赋时，按钮做微小的 scale pop (0.05, 0.1s) + 边框颜色从默认过渡到绿色或取消 |
| **P2-6** | 动态动效参数系统 | 全局 Tween 辅助层 | 中 | 根据控件尺寸自动计算旋转角/scale/时长。可以用一个静态工具函数实现，不需要框架 |

## 五、给程序团队的几句话

我再说一遍：**不写代码，但我把方向说清楚。**

1. **不要从零造轮子。** Godot 4.x 的 `create_tween()` 已经非常好用——DamageNumber 里那个并行 Tween 是正确方向。在这个基础上封装，不要引入第三方库。

2. **动效参数集中管理。** 不要把 `0.25`、`0.12`、`1.05` 这些魔法数字散落在各个文件里。建一个全局动效配置常量表，颜色、时长、scale 幅度、stagger 间隔都放在一起。这样我改一个数字不用翻遍所有文件。

3. **性能上不要担心。** 这些都是 UI 层的 Tween，不在 `_process()` 里逐帧运算。12 个以下的并行 Tween 根本不会有性能影响。但要注意：战斗中不要同时触发超过 5 个 UI Tween——不是性能问题，是视觉噪音问题。

4. **先做 P0-1。** PanelManager 的 show/hide 过渡改好了，整个游戏的 UI 基因就变了。那个改动可能不到 30 行，但价值超过后面所有加起来。

5. **所有动效必须可跳过。** 如果玩家在 0.3 秒内连续按下两个按键（比如快速跳过对话、快速选升级），动效应该被加速或直接跳到结束状态。不要让动效变成"等动画播完才能操作"的阻碍——这个平衡是动效设计最难的部分。

## 六、一个美术总监的碎碎念

我在金翅鸟壁画前站了三个小时，不是因为壁画有多复杂——是因为一笔飞白里藏着画师整个人的力道。

UI 动效也是一样的。

一个击杀数字的弹跳如果幅度大了 0.05，就不是"有分量的打击"而是"轻浮的弹窗"。一个面板淡入如果快了 0.1 秒，就不是"从黑暗中浮现"而是"屏幕闪了一下"。

**所有动效参数的最终校准，不是在编辑器里看 Tween 曲线——是眯着眼睛，身体往后靠，感觉这个动效的"重量"对不对。**

暗色主题的 UI 动效，分量比速度重要。克制比花哨重要。留白比填充重要。

一个按钮的 hover 旋转如果让按钮的角落移动了超过 3 个像素，那就不是"回应我的触碰"，而是"按钮在发抖"。

诸君，写代码的时候，记得把 Tween duration 设好了以后，**先减 0.02 试试**。很多时候，最好的动效是你刚刚好感觉不到它在动的那个临界点。

—— 杨奇
