## 问题 1：场景迷雾

### 重新诊断

之前我说"CanvasLayer 是错的，应该放世界层"。错了。

真正的问题在调用链上：

1. `MapSystem.update()` 负责根据玩家位置更新 `_fog_grid`（把 `false` 改成 `true`）
2. 谁在调用 `MapSystem.update()`？只有 `_toggle_map()`（按 M 键时）
3. `_update_scene_fog()` 每 0.5s 运行，读取 `_fog_grid` 生成雾纹理
4. 但 `_fog_grid` 在 gameplay 期间从未被更新——全是 `false`
5. 所以 `get_scene_fog_texture()` 返回的纹理全是黑色半透明
6. "已探索区域也一团黑"——因为探索数据从未被记录

**根因一句话**: `MapSystem.update()` 只在打开地图时调用。gameplay 期间的探索从未被写入 `_fog_grid`。

CanvasLayer 本身不是问题。分辨率（72×54 拉伸）是视觉质量问题，不是逻辑 bug。

### 选定方案: 修复数据更新频率

**不删除任何代码。不改变渲染架构。不移动雾到世界层。**

**Step 1** — 在 `game_manager.gd` 的 `_process()` 中，雾更新前先更新探索数据:

**当前代码（第 762-765 行）**:
```gdscript
if _scene_fog_timer > 0.5:
    _scene_fog_timer = 0.0
    _update_scene_fog()
```

**改为**:
```gdscript
if _scene_fog_timer > 0.5:
    _scene_fog_timer = 0.0
    _update_map_from_system()   # 先更新 _fog_grid 探索状态
    _update_scene_fog()         # 再用最新数据生成雾纹理
```

**效果**: 已探索区域的 cell 会在 0.5s 内被标记为 `true`，雾纹理中对应像素变透明。已探索区域不再一团黑。

**Step 2** (可选，解决"远处敌人可见") — 在 `_update_scene_fog()` 末尾加敌人可见性控制:

在 `MapSystem` 中暴露 `get_fog_grid()`:
```gdscript
## 暴露探索网格供外部查询
func get_fog_grid() -> Array:
    return _fog_grid
```

在 `game_manager.gd` 中:
```gdscript
func _update_scene_fog() -> void:
    # ... 现有纹理生成逻辑不变 ...
    
    # 根据雾状态控制敌人可见性
    var ms: Node = $"../HUDLayer/MapSystem"
    if ms and ms.has_method("get_fog_grid"):
        var grid: Array = ms.get_fog_grid()
        if not grid.is_empty():
            for c in enemies.get_children():
                if not c is CharacterBody2D: continue
                var cx := int(c.global_position.x / 80.0)
                var cy := int(c.global_position.y / 80.0)
                if cx >= 0 and cx < grid.size() and cy >= 0 and cy < grid[0].size():
                    c.visible = grid[cx][cy]
                    if not grid[cx][cy]:
                        c.set_process(false)
                        c.set_physics_process(false)
```

### 风险评估
- **极低** — 只加了一行函数调用 + 一个可选的敌人遍历。`_update_map_from_system()` 已存在且无副作用。
- **注意**: `_update_map_from_system()` 内部会更新 M 地图面板的元素（PlayerMarker 位置等）。地图面板关闭时这些更新无害——下一帧可见性检查会跳过隐藏的 Control。
- **性能**: 每 0.5s 一次 `MapSystem.update()`（遍历玩家周围 25 个 cell）——可以忽略。

### 分辨率优化（下一轮，非本次修复）
- 当前: 72×54 ImageTexture，STRETCH_SCALE 到 1920×1080，最近邻过滤，方块感严重
- 计划: 生成 360×270 纹理 + 设置 `texture_filter = CanvasItem.TEXTURE_FILTER_LINEAR`，平滑过渡
- 或: 用 shader 在 cell 边界做 2-3 像素渐变消除方块

## 问题 2：面板优先级

### 重新诊断

之前我说"没有人用 PanelManager"、"GameState 应该驱动面板关闭"。方向偏了。

真正的 bug 在 `_trigger_stage_dialogue()` 和 `_show_upgrade_panel()` 之间：

1. 升级面板弹出 → `GameState.set_state(State.PAUSED)` + `get_tree().paused = true`
2. 阶段完成触发对话 → `GameState.set_state(State.PAUSED)` — **状态没变，因为已经是 PAUSED**
3. 对话直接在升级面板上面显示
4. 对话结束 → `GameState.set_state(State.PLAYING)` → 游戏恢复
5. 升级面板的 `_is_paused` 仍然是 `true`，`visible` 仍然是 `true` — **残留**

**根因一句话**: 两个暂停源（升级、对话）共用 `State.PAUSED`。状态机无法区分它们。对话结束时状态回到 `PLAYING`，但升级面板不知道发生了什么。

`game_state.gd` 第 7 行已经定义了 `DIALOGUE` 状态。第 25 行 `State.DIALOGUE: get_tree().paused = true` 已经实现了自动暂停。一切都已经在那里了——只是对话代码没用它。

### 选定方案: 对话用 State.DIALOGUE + 弹出前关闭升级面板

**Step 1** — 修复 `game_manager.gd` 的 `_trigger_stage_dialogue()`:

**当前代码（第 1237-1251 行，错误）**:
```gdscript
GameState.set_state(GameState.State.PAUSED)  # 错误：和升级面板用同一个状态
get_tree().paused = true                       # 多余：State.PAUSED 已经自动暂停
dlg.show_dialogue(msgs)
await dlg.dialogue_finished
get_tree().paused = false                      # 多余：State.PLAYING 已经自动恢复
GameState.set_state(GameState.State.PLAYING)
```

**修复后**:
```gdscript
# 进入对话前关闭升级面板（如果开着）
if level_up_panel.visible:
    _finish_upgrade_chosen()

GameState.set_state(GameState.State.DIALOGUE)  # 用 DIALOGUE 而非 PAUSED
dlg.show_dialogue(msgs)
await dlg.dialogue_finished
GameState.set_state(GameState.State.PLAYING)
```

**效果**:
- `State.DIALOGUE` 自动设置 `get_tree().paused = true`（已在 game_state.gd 第 25 行实现）
- `State.PLAYING` 自动设置 `get_tree().paused = false`（第 27 行）
- 不再需要手动管理暂停
- 如果对话期间升级面板试图弹出——它的 GameState 检查会发现当前状态是 DIALOGUE 而非 PLAYING（但目前没有这个检查路径，`_on_level_up_available` 通过 `await` 触发，在对话期间不会执行）

**不需要**: 修改 `game_state.gd` 的 `set_state()`。不需要 `_close_panels_for_state()`。不需要 PanelManager 集成。不需要 ModalStack。

**为什么不需要更复杂的状态转移逻辑**: 在这个游戏里，唯一可能在 `PAUSED`（升级）期间弹出的只有对话（阶段完成触发）。地图面板（M 键）在暂停期间玩家不会按。结算面板在游戏结束时——那时候没有升级面板。只有一个冲突对。

### 风险评估
- **极低** — 改了一个函数的前三行。`State.DIALOGUE` 的暂停/恢复逻辑已经在 `game_state.gd` 中实现且经过测试（第 25 行 `State.DIALOGUE: get_tree().paused = true`）。
- **边界情况**: 如果对话在升级面板动画（Tween）进行中触发，`_finish_upgrade_chosen()` 需要正确处理。当前它只是设 `visible=false` 和恢复 `process_mode`——这在 Tween 进行中也安全。
- **`_pending_boss_upgrades`**: 如果 Boss 战中升级被延迟，对话关闭后不会自动重新弹出升级面板。这是一个预存问题，不是本次修复的范围。

## 两个问题的关联（修订）

之前的类比（"数据在 A 层，渲染在 B 层" / "调度在 A 系统，执行在 B 系统"）过度抽象了。

**真正的共同模式**: **已有的系统写了正确的逻辑，但在关键时刻没有被调用。**
- `MapSystem.update()` 写了探索更新逻辑——gameplay 期间没被调用
- `GameState.State.DIALOGUE` 定义了对的状态——对话代码用了 `PAUSED` 而不是它
- `game_state.gd` 自动暂停/恢复逻辑已实现——`_trigger_stage_dialogue()` 手动又做了一遍

**修复原则（修订）**: 不是"让已有系统做它被设计来做的事"。而是"检查调用链——确保数据在被读取之前已经被更新，确保状态枚举被正确使用"。
