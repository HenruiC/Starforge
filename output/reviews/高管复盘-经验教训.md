# 高管复盘 — 从迷雾和面板问题中学到了什么

> 复盘日期：2026-05-19
> 背景：迷雾系统经历 3 次失败尝试（UI TextureRect 覆盖→全屏拉伸→世界对齐），最终用 TileMap 世界层方案解决；面板优先级同样经历失败（panel_manager 加了队列但对话绕过它），最终用 GameState.State.DIALOGUE 驱动方案修复。
> 共同模式：已有的系统写了正确的逻辑，但在关键时刻没有被调用。

---

## 小岛秀夫（制作人）— 创意与架构视角

我们写了一部剧本，拍好了所有镜头，然后把它们锁在了不同的保险柜里，保险柜上贴的标签不一样——有人贴"PAUSED"，有人贴"DIALOGUE"。

这不是技术问题，是信任问题。我们设计了 MapSystem —— 一个有自己数据模型、有 update() 方法的自治系统。然后 game_manager 决定：我不要你的 update()，我自己开一个 `_build_map_viewport()`，用 10,800 个 ColorRect 重写一遍。GameState 定义了 `State.DIALOGUE`，而对话代码说：我不信任这个枚举，我要手动 `get_tree().paused = true`。

"数据离它的消费者有多远"——答案是：零距离。MapSystem 和 GameState 都是 Autoload，全局可达。错不在距离，在调用方不信任被调用方。系统之间的契约写了但没人读。下次设计数据流时，不仅要写契约，还要在每一个消费者入口放一行断言：我使用的是系统提供的接口，不是我自己造的替代品。

---

## 宫崎英高（战斗策划）— 游戏设计视角

迷雾是不是"视觉效果"？不。迷雾是信息剥夺。信息剥夺是最古老、最有效的恐怖游戏机制。

如果我们从一开始就把迷雾定义为 gameplay 机制——"敌人存在于迷雾之中，但玩家看不见它们"——那么第一次 code review 就会有人问："敌人可见性检查在哪？你凭什么说玩家看不见它们？"然后我们会在五分钟内发现 `_fog_grid` 全是 `false`。

但我们没有问。因为我们把迷雾当成了"美术特效"——一个覆盖在屏幕上的半透明黑色纹理。"好看就行。"错了。在游戏设计里，每一个视觉效果如果和 gameplay 有关，就必须作为机制来设计、来测试、来防御。迷雾消散的 tile 不是美术资源，是信息 token。敌人出现在未探索区域不是"视觉瑕疵"，是机制失效。下一次，任何影响玩家决策的视觉效果，必须从机制文档开始，不是从美术需求开始。

---

## 马斯克（主程）— 工程视角

同一个问题修了三次。根因不是技术选型错误——四个方案里至少两个可行。根因是**每次修复只修了症状链的第一环，没有追到根节点的数据源。**

第一轮（TextureRect）：发现"雾不动"→ 认为是渲染层的毛病 → 换覆盖方式。没问：雾的数据从哪来？
第二轮（全屏拉伸）：发现"尺寸不对"→ 改拉伸参数。没问：分辨率为什么是 72×54？
第三轮（世界对齐）：发现"坐标错位"→ 改坐标系。终于有人问了一句：`_fog_grid` 里到底存了什么？

答案是：全是 `false`。因为在 gameplay 期间 `MapSystem.update()` 从未被调用。这个答案在第一轮就该被找到——只要有人从根节点（数据源）开始 debug，而不是从叶子节点（屏幕上的黑色区域）开始。

方案 A vs 快速修复的决策公式：**如果 bug 的症状有多个独立表现（雾不动 + 分辨率错 + 坐标歪），说明根因在更深处。此时快速修复一定是浪费。** 三个症状 = 一个根因。修根因的 1 行代码，省了三次修症状的 50 行。

## 陶德·霍华德（制作人）— 玩家体验视角

玩家不会看 `_fog_grid`。玩家只会看到"我都走到体育馆了，雾还是黑的"——然后觉得地图系统坏了。玩家不会看 `game_state.gd` 的状态转移表。玩家只会看到"升级面板还开着，对话突然盖上去，然后两个都关不掉"——然后按 Alt+F4。

这两个 bug 的玩家体验破坏力远超它们的代码量（合计修复 4 行）。说明一个残酷的事实：**玩家接触到的每一帧画面都是集成测试，而我们没有在集成层做测试。**

下一个功能怎么避免？一条铁律：每个系统完成时，不只需验证它的内部逻辑正确，还需要验证它在"玩家真实会走的主路径"上正确。MapSystem 的内部逻辑是对的——`update()` 写得很好。但它从未在玩家走路时被调用。这是集成断裂。下次，任何被玩家直接感知的系统（迷雾、面板、对话、战斗 HUD），交付时必须回答一个场景问题："玩家从出生走到 Boss 间，这个系统在每一段路上表现如何？"不是"单元测试过了"，是"玩家旅程过了"。

## 库克（主QA）— 质量视角

如果当时有两份测试用例——"迷雾能挡住未探索区域的敌人吗？"和"升级面板打开时触发阶段对话会发生什么？"——这两个 bug 会在各自系统的首次验收测试中被发现，耗时不超过 15 分钟。

我们没有这两份用例。因为我们在写测试用例时问的是"这个系统应该做什么？"——而不是"这个系统承诺了什么？两者不一样。"应该做什么"是期望，"承诺了什么"是契约。MapSystem 承诺了 `update()` 方法输出正确的 `_fog_grid`。但我们没有测"gameplay 期间 `_fog_grid` 是否被正确消费"。面板系统的 GameState 承诺了 `State.DIALOGUE` 状态自动暂停。但我们没有测"对话是否使用了正确的状态枚举"。

**测试用例必须针对系统的契约边界编写。** 契约的边界不在系统内部——在"谁调用了这个系统"那一行代码上。下次，每个核心系统验收时，至少两个跨边界测试用例：(1) 数据源正确更新后，消费者是否正确收到了？(2) 两个系统在同一时刻触发时，状态机能否正确排序？

## 下次怎么做 — 三条可执行的规矩

### 规矩 1：调用链行走（Call Chain Walk）

任何系统标记为"完成"之前，必须做一次端到端的调用链行走：从数据生产者（MapSystem.update() / GameState.set_state()）逐行追踪到数据消费者（FogOverlay.texture / get_tree().paused），确认中间没有断开的环节。行走时不能假设"那个函数应该被调了"——必须在代码中找到具体的调用点和触发频率。写一行注释记录行走路径：`# Call chain: _process() → _update_map_from_system() → MapSystem.update() → _fog_grid modified → _update_scene_fog() reads it`。如果写不出这行注释，说明走不通。

### 规矩 2：契约测试优先于功能测试

功能测试回答"这个函数做了它该做的事吗？"契约测试回答"调用方按照约定使用这个函数了吗？"两者不同。P0 bug 从不来自功能层面——`MapSystem.update()` 的功能从来没问题。P0 bug 来自契约层面——gameplay 期间没人调用它。以后每新增一个公共方法/枚举/信号，必须至少写一个契约测试用例（可以只是文档中的一行检查项，不需要自动化）："谁在什么情况下调用它？调用方使用的是这个方法还是绕路写了一份替代逻辑？"

### 规矩 3：三症状 = 一根因

当同一个功能域出现三个或以上独立症状（迷雾：位置错 + 尺寸错 + 探索不更新），停止修症状。回到数据源重新追踪。在数据源确认正确之前，不许提交任何 UI 层的修复。这条规矩不需要新工具——只需要在 code review 时问一句："这个问题的数据从哪来？源头确认过吗？"如果答案不是"确认过"，reject。
